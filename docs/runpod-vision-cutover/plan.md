# Plan: RunPod Vision Cutover

## Approach

`api/src/lib/vision.ts` already treats Tier 2 as "any OpenAI-compatible endpoint" via `VISION_API_BASE`/`VISION_API_KEY`/`VISION_API_MODEL`, and `runVisionManaged`/`processImage` already fail open to per-image errors — so the cutover itself is a desktop `.env` edit, not a code change, *provided calibration proves the RunPod-served model actually answers*. The actual engineering work is three things that env edit doesn't cover: (1) a feasibility spike confirming the intended serving stack loads the intended model at all; (2) a cost-safety watchdog for RunPod dedicated pods, which have no built-in idle-stop (proven cost incident in the sibling algo-corpus project), reusing that project's proven `podStop`-on-idle mechanism as-is rather than inventing a new one; and (3) a calibration pass that runs the RunPod backend over the frozen ADR-0010 reference set and measures backend agreement rate, per-image latency (steady-state and cold-start separately), and throughput before RunPod is trusted as the production default — because RunPod serverless throughput for vision is an open, unproven problem in the vLLM community, mirroring the exact serverless-throughput failure algo-corpus already hit for its (text) extraction cascade.

Calibration must also explicitly rule out a known failure mode already hit and fixed once in this codebase: `vision.ts` documents (around `LOCAL_GATE_MAX_TOKENS`) that qwen3-vl is a reasoning-tuned model family that spends its token budget on chain-of-thought before answering — at a low token budget, `content` came back empty on every call in production, silently defeating the local gate until `/no_think` + a larger `num_predict` fixed it. That fix was applied only to `runLocalGate`/`LOCAL_GATE_SYSTEM`. `runVisionManaged` and `VISION_SYSTEM_PROMPT` — the exact path this cutover routes Tier 2 through — carry no such guard today. See Risk areas and Design decisions below; this is in scope for calibration to test and, if it reproduces, in scope to fix — not a follow-up.

Both deployment modes (serverless endpoint / dedicated-pod proxy) are supported purely by the value of `VISION_API_BASE`, so the calibration result is applied with an env change alone, once the spike and calibration above have passed.

## Architecture

```
                         estate-scraper desktop (service user: estate-scraper)
                         ┌───────────────────────────────────────────────┐
weekly Scan (systemd/    │  npm run scan                                 │
LaunchAgent trigger) ───►│    processSalesStream()                       │
                         │      Tier 0 (pHash/quality, CPU)              │
                         │      Tier 1 (SigLIP embed rank, post-scan)    │
                         │      Tier 2 → runVisionManaged()              │
                         │        POST {VISION_API_BASE}/chat/completions│
                         └───────────────┬───────────────────────────────┘
                                         │ OpenAI-compatible wire (unchanged)
                                         ▼
                         ┌───────────────────────────────┐
                         │  RunPod                        │
                         │  serverless endpoint  OR        │
                         │  dedicated pod + proxy URL       │
                         │  (vLLM, Qwen-VL, worker-vllm)   │
                         └───────────────┬───────────────┘
                                         │ RunPod GraphQL (podStop)
                                         │ (dedicated-pod mode only)
                         ┌───────────────┴───────────────────────────────┐
                         │  systemd timer (every 15 min, own service)     │
                         │    runpod_vision_watchdog.sh                   │
                         │      reads api/data/scan-state.json's `phase` │
                         │      idle := phase !== "analyzing"            │
                         │      if idle: sleep 60s inline, recheck once  │
                         │      → RunPod GraphQL podStop on RUNNING pod  │
                         │        whose name is an exact/strict-prefix   │
                         │        match for estate-scraper's own pods    │
                         │        (never a substring — see Design        │
                         │        decisions)                             │
                         └───────────────────────────────────────────────┘
```

The watchdog is a standalone process, independent of the Scan schedule (systemd timer, not triggered by scan start/end), matching requirement 5. It reads the same `scan-state.json` the API already writes/serves at `/api/status` — no new signal is invented, but the *field* it reads is `phase`, not `running`. `running` is true for the entire Scan (scraping → analyzing → embed pass), including Tier 0/1 work (pHash, CPU-side quality gate, SigLIP ranking) that never touches the vision backend at all — a Scan that's still 20 minutes from its first Tier-2 call would otherwise read as "busy" and hold a dedicated pod up the whole time it's just scraping. `phase` (`"idle" | "scraping" | "analyzing" | "done"`, `api/src/scan/state.ts`) already distinguishes this: only `"analyzing"` covers vision calls, so idle is `phase !== "analyzing"`. Because the watchdog runs under the same `estate-scraper` service user as the app (not a separate account), it reads the state file directly off disk rather than making an authenticated HTTP call to `/api/status` — simpler than minting a bearer token for a cron job, and the file is already the source of truth `readScanState()` itself falls back to on a torn read.

Calibration is a one-time (per candidate model/config), offline step, gated on a feasibility spike (see Design decisions) that runs first: point `VISION_API_BASE` at the candidate RunPod endpoint, re-run the reference-pass code path (`npm run scan -- --reference <path>`) over the same or a representative subset of the frozen reference pass's sales, then diff the two reference-pass JSON files with a new script.

## Data model

One additive change, not zero: a composite index on `(sale_id, vlm_model)` on both `findings` and `finding_items`. Acceptance criterion 1 ("query rows by vlm_model") has no index support today — `idx_findings_sale_id`/`idx_finding_items_sale_id` (`api/src/db/schema.ts` lines 71–72, 102–103) only cover `sale_id` alone, so a query filtering or grouping by `vlm_model` (e.g. "how did the RunPod-origin rows compare to the Gemini-origin rows for this sale") is a full table scan. Cheap to add now while the corpus is still Gemini-only; expensive to retrofit once it spans multiple VLM generations across tens of thousands of rows. Both are safe, non-hazardous `CREATE INDEX` statements — no `ALTER TABLE` column rewrite, no lock contention with existing rows, no backfill:

```ts
// findings
idxSaleVlmModel: index("idx_findings_sale_vlm_model").on(t.saleId, t.vlmModel),
// finding_items
idxSaleVlmModel: index("idx_finding_items_sale_vlm_model").on(t.saleId, t.vlmModel),
```

Generated via `drizzle-kit generate` into a new `api/drizzle/` migration, applied by the existing `runMigrations()` call at Scan startup (`api/src/db/index.ts`) — no new migration-running mechanism needed.

`findings.vlmModel`/`findings.promptVersion` and `finding_items.vlmModel`/`finding_items.promptVersion` already exist as nullable text columns per ADR-0016, and `api/src/scan/persist.ts` already stamps every insert with `activeVlmModel()` (returns `VISION_API_MODEL` whenever `VISION_API_BASE` is set) and `PROMPT_VERSION` — confirmed by reading `persist.ts` lines 166–198. Pointing `VISION_API_MODEL` at a RunPod model string requires zero further migration; RunPod-origin rows are distinguishable from Gemini-origin rows by that column alone, exactly as ADR-0016 intends for a VLM swap (not a corpus migration, unlike the frozen embedding model). No CHECK/NOT NULL constraint is being added on `vlm_model` — see Design decisions ("Rejected findings").

## API / interface contract

None. This cutover is env-var config on the desktop's production `api/.env`, plus a new standalone watchdog script + systemd units + a calibration script — no new HTTP endpoint, CLI flag on the app itself, or UI surface. The existing `--reference` flag on `npm run scan` (already implemented) is reused as-is for calibration, not extended.

## Integration points

- `api/.env.example` — add a `RUNPOD_API_KEY` line (currently only aspirationally documented in the README config table, not present here) with a comment distinguishing it from `VISION_API_KEY`: `VISION_API_KEY` is the Bearer token `runVisionManaged` sends to the inference wire; `RUNPOD_API_KEY` is the account-level key the watchdog uses for RunPod's GraphQL management API (`podStop`). Placeholder value only, per the non-functional security requirement. The existing Gemini/RunPod `VISION_API_BASE` example block is already correct and needs no change (verified — both examples already present, satisfying acceptance criterion 8).
- Desktop production `api/.env` (deploy-time, not repo-committed, path `/home/estate-scraper/estate-scraper/api/.env` — confirmed against `scripts/deploy-remote.sh`'s `DEPLOY_HOME=/home/$DEPLOY_USER` and rsync target, **not** `~/.env`) — set `VISION_API_BASE`/`VISION_API_KEY`/`VISION_API_MODEL` to the calibrated RunPod values, and add `RUNPOD_API_KEY`. **There is no automated credential-injection mechanism in this repo** — `deploy-remote.sh` explicitly excludes `.env` from every rsync (`--exclude='.env'`, both the build-local sync and the remote install step). This is, and remains, a manual one-time SSH edit to that file on the desktop, same as how the operator must already be setting `VISION_API_KEY`/`ORACLE_API_KEY` today. Reverting to Gemini is unsetting/rewriting these vars by hand and restarting the service — no deploy, no tooling either direction.
- `scripts/runpod_vision_watchdog.sh` (new) — bash watchdog reusing `~/dev/algo-corpus/scripts/runpod_ingest_watchdog.sh`'s real, proven mechanism: a single-execution inline check, not a persistent cross-invocation state file. On each systemd timer firing: load `.env`; read the idle signal; if busy, log and exit; if idle, `sleep 60`, read the signal again, and only act if it's still idle on that second read within the same process. No `/tmp` counter file, no dependency on state surviving between separate timer firings or across a reboot — this matches algo-corpus's `busy()`/recheck shape exactly rather than an invented "two consecutive reads separated by a 5-minute cooldown" scheme that would need its own persisted counter. The idle signal itself is estate-scraper-specific: `phase !== "analyzing"` read directly off `api/data/scan-state.json` on local disk (no HTTP call, no auth), vs. algo-corpus's remote LightRAG HTTP status poll (`/documents/status_counts`) — the one real adaptation, forced by the different source of truth, not by a different mechanism. A staleness guard the algo-corpus version didn't need: if `phase === "analyzing"` but `startedAt` is older than a configurable `WATCHDOG_MAX_SCAN_HOURS` (default 6 — well over a normal budgeted Scan's runtime, well under the ADR-0010 full reference pass's ~23h), treat it as idle anyway, so a Scan process that died without calling `writer.finish()` (SIGKILL, OOM, host reboot) doesn't wedge the pod on forever — this is what actually satisfies requirement 5's "pod left running by a failed or aborted Scan is still caught" for the non-graceful-crash case (the graceful-failure case already flips `running: false` via `markScanFailed`/`writer.finish(failed=true)` today). GraphQL failures fail loud, not silent: unlike algo-corpus's own Python parser (`d.get('data', {}).get('myself', {}).get('pods', []) or []`, which treats an auth error, a malformed response, and "genuinely no pods" identically), this watchdog checks explicitly for a curl failure or a top-level `errors` key in the GraphQL response and, on either, sends a distinct ntfy alert ("watchdog could not read pod list — nothing was touched") and logs it as an error, separately from the normal "still busy" and "nothing to stop" no-op paths — collapsing a management-API failure into a silent no-op is the exact runaway-billing scenario this script exists to prevent (see Risk areas). Pod-name matching uses an exact match or strict prefix (`estate-scraper-vision-*`), never a loose substring — see Design decisions. Logs to `runpod-watchdog.log`, optional `NTFY_URL` alert on a stop.
- `systemd/estate-scraper-runpod-watchdog.service` and `systemd/estate-scraper-runpod-watchdog.timer` (new, repo root, following algo-corpus's `systemd/algo-corpus-runpod-watchdog.service`/`.timer` shape) — oneshot service + timer (`OnCalendar=*:0/15`, `RandomizedDelaySec=60`, `Persistent=true`), with an explicit `WorkingDirectory=` and an absolute `ExecStart=` path (e.g. `WorkingDirectory=/home/estate-scraper/estate-scraper`, `ExecStart=/home/estate-scraper/estate-scraper/scripts/runpod_vision_watchdog.sh`), matching the real algo-corpus unit's shape line-for-line on that point. One deliberate deviation from the algo-corpus template: estate-scraper's own production app service is already deployed as a **user-level** systemd unit (`systemctl --user restart estate-scraper`, per `scripts/deploy-remote.sh`), not algo-corpus's system-level (`User=`/`Group=`, `WantedBy=multi-user.target`) pattern — so the watchdog units drop `User=`/`Group=` (redundant under a user-level instance), use `WantedBy=default.target`, and install under the estate-scraper service user's own systemd user instance, for consistency with how this repo already deploys.
- `api/eval/calibrate-runpod.ts` (new) — the calibration script. Takes `--baseline <path>` (the frozen Gemini reference-pass JSON, an array of `ReferenceRecord` as already written by `npm run scan -- --reference <path>`) and `--candidate <path>` (a RunPod reference-pass JSON produced the same way, with `VISION_API_BASE` temporarily pointed at the candidate endpoint over the same or a representative subset of sales — not a full $22 reference re-run per candidate). Matches records by `imageUrl` (the stable key across both files), imports the production `hasFindings()` from `../src/vision/index.js` so scoring logic is never duplicated/drifted, and reports:
  - **backend agreement rate** as `matched-hasFindings / baseline-hasFindings` (fraction of the baseline's flagged images the candidate also flags). Deliberately *not* called "recall@K" — ADR-0010 already uses that term for a top-K ranking-cutoff metric (Tier 1's rank vs. the frozen reference pass); this feature's metric has no K/cutoff, it's a full-set agreement rate between two backends over the same image set, and reusing ADR-0010's name for a structurally different measurement would be confusing across the two docs.
  - average/p50/p95 per-image latency computed over all images **except the first**, plus a separately-reported **cold-start latency** (the first image's `durationS` alone) — a serverless 32B-class vision model likely has non-trivial cold-start time that a blended average would hide, and the plan's "serverless first" justification (Technology choices) never weighed it until now.
  - throughput, accepted as an operator-supplied `--elapsed-s` flag (wall-clock seconds for the candidate run) rather than summed per-image durations — summing would overstate wall-clock time under `VISION_WORKERS` concurrency.
  - a required **item-level spot-check**: before the calibration decision is recorded, a fixed sample of the **first 30 disagreements** between baseline and candidate `hasFindings` (or all disagreements if fewer than 30) must be manually read side-by-side and judged for whether the candidate is actually wrong, or whether `hasFindings()`'s Gemini-tuned heuristics (junk-line detection, verbose-prefix rejection, a 1400-char cap) are misjudging Qwen-VL's different phrasing style. 30 is small enough to review in one sitting, large enough to catch a systematic heuristic mismatch rather than one-off noise. "Spot-check by eye, no number" is not sufficient — the number is 30, and it's reviewed before the number goes in `calibration-results.md`, not after.

  Add an npm script `"calibrate:runpod": "tsx eval/calibrate-runpod.ts"` in `api/package.json`.
- `api/src/vision/index.ts` and `api/src/scan/index.ts` — two additive fields onto the reference-pass code path, both scoped to `ReferenceRecord`/the `image_result` event only (they do not touch `runVisionManaged`, `processImage`'s error-handling contract, or the normal-scan write path, so requirement 1's "no code changes... for the RunPod cutover itself" and requirement 9's fail-open error contract are unaffected):
  - `durationS` — the `image_result` event (reference mode) currently forwards `response`/`error`/`positionIndex`/`total`/`hasFindings` but drops the `durationS` that `processImage`'s `ImageResult` already computes (see `refResults` loop, `vision/index.ts` ~line 742). Forward it onto both the event and `ReferenceRecord` (`scan/index.ts`) so it lands in the reference-pass JSON — needed for the latency/cold-start figures above.
  - `backend` — `ReferenceRecord` (`scan/index.ts` lines 67–77) today carries no model or backend provenance at all: nothing in the reference-pass JSON records whether a given image was actually served by Gemini, RunPod, or a silent fallback to local Ollama. This is the concrete failure mode found in review: `run-scan.sh` explicitly sources `.env` (`set -a; [ -f .env ] && source .env; set +a`) before every real Scan, but a one-off calibration command run by hand from an SSH session has no such guarantee — this codebase has no `dotenv` auto-loading anywhere, only that one manual `source`. If an operator runs `--reference` against a candidate RunPod endpoint but forgets to export/source the temporary env vars into that shell, `VISION_API_BASE` reads as unset, `runVisionManaged` never fires, and the scan silently falls through to the local Ollama model — producing a "candidate" JSON that is actually a mislabeled Ollama run, with nothing today to catch it. Add `backend: string` to `ReferenceRecord`, populated from `activeVlmModel()` (`vision.ts`) at record-construction time, so `calibrate-runpod.ts` can assert every record in a candidate file actually carries the intended RunPod model string before trusting the comparison. `CALIBRATION.md` (below) must instruct the operator to run calibration the same way `run-scan.sh` loads config (`set -a; source .env; set +a`, or an explicit `export`) before invoking `npm run scan -- --reference`, not just setting vars for the current shell command alone.
- `README.md` — update the Stack table's Tier 2 row and Configuration table if the calibrated model/deployment mode differs from what's currently documented (`qwen3-vl:30b` / `Qwen3-VL-32B`, RunPod serverless); add a one-line pointer to this cutover's calibration doc next to the existing "Calibration" section.
- `docs/adr/0010-budget-bounded-runpod-cascade.md` — no decision change (Tier 2 backend choice was already RunPod-shaped in this ADR's own text), but worth a short addendum noting the backend is now Gemini-or-RunPod via env, calibrated per this cutover, since the ADR's prose currently reads as if RunPod were already the live Tier-2 backend. Also worth a one-line cross-reference disambiguating the ADR's own `recall@K` (Tier-1 ranking-cutoff metric) from this cutover's differently-named backend agreement rate, since both docs now sit next to each other.
- `docs/runpod-vision-cutover/CALIBRATION.md` (new, created by step 11) — operator-facing calibration procedure documenting: the feasibility spike (below) as step zero; how to provision a RunPod endpoint; the `.env`-sourcing requirement above; how to run the reference scan; how to interpret backend agreement rate, latency/cold-start, and the 30-item spot-check; and the explicit teardown step if calibration is abandoned (see Risk areas — pod deprovisioning).
- `docs/runpod-vision-cutover/calibration-results.md` (new, written after the calibration run by step 14) — the durable record acceptance criterion 3 requires: backend agreement rate, per-image latency (steady-state and cold-start), throughput, the spot-check outcome, and the deployment-mode/model decision that followed from them.

## Technology choices

- **A feasibility spike comes before any of the systemd/calibration/watchdog machinery above is built.** The research sweep's GPU/config guidance and the model choice below were both gathered against Qwen2.5-VL forum threads and `worker-vllm` examples — none of it confirms `runpod-workers/worker-vllm` actually loads and serves the specific `Qwen3-VL-32B-Instruct` checkpoint this plan names. The spike is: stand up one endpoint (serverless, smallest viable config) with that exact checkpoint and confirm it returns a real completion for one image. If it fails to load or serve, the fallback path is, in order: (a) the equivalent Qwen2.5-VL-32B/72B checkpoint the research was actually validated against, (b) another vLLM-supported vision-instruct checkpoint of comparable size, recording whichever substitution was made and why in `calibration-results.md` before any further calibration work proceeds. Everything downstream (calibration, the watchdog, the systemd units) is unaffected by which checkpoint wins — they're all env/config, not code — so the spike is purely about not building the rest of this plan against a model that never loads.
- **Attempt serverless first, dedicated pod as the calibrated fallback.** ADR-0010's cost model assumes both cascade tiers scale to zero; serverless is the cheaper default to try first and needs no watchdog. The research sweep flags RunPod serverless vision throughput as unproven (same failure shape as algo-corpus's ~6 chunks/min serverless bottleneck for its LLM cascade) — so this is a hypothesis to calibrate, not an assumption to ship on. The watchdog is built regardless of which mode wins, since requirements 3–5 mandate it unconditionally and a dedicated-pod fallback needs it ready on day one, not built reactively after a bill surprise — and, per Design decisions below, must be verified working *before* a dedicated pod is ever provisioned, not after.
- **`runpod-workers/worker-vllm`** as the serverless template (459★, RunPod's own official image) — named directly in the research sweep, already the pattern several Qwen-VL-on-RunPod examples build from; reusing it avoids hand-rolling a vLLM Docker image. Checkpoint compatibility is confirmed by the feasibility spike above, not assumed.
- **Qwen3-VL-32B-Instruct** — already the model estate-scraper's own README/`.env.example` document for this slot; no reason to switch families mid-cutover unless the feasibility spike forces a substitution. Calibration must explicitly test for the reasoning-mode empty-response bug already hit and fixed for the local gate (see Approach and Risk areas): send a handful of calibration images through the candidate endpoint and confirm `content` is non-empty and not truncated mid-thought. If the bug reproduces against the RunPod-served checkpoint, the fix is in scope for this cutover, not deferred — whichever of the following actually works against this serving stack: a `/no_think`-equivalent prefix prepended to `VISION_SYSTEM_PROMPT`; vLLM's `chat_template_kwargs: {"enable_thinking": false}` request parameter, if `worker-vllm` exposes it for this checkpoint; or, failing both, `<think>...</think>` tag stripping added to `runVisionManaged`'s response parsing before the content is handed to `hasFindings()`/persisted.
- **`--limit-mm-per-prompt image=1`** on the vLLM launch args — `processImage` sends exactly one image per chat completion call (never a multi-image batch), so this vLLM multimodal flag (required per the vLLM serving docs found in research) should be set to the minimum, both to satisfy vLLM's explicit-limit requirement and to keep the KV-cache memory profile as small as possible for a 48GB-class GPU.
- **48GB-class GPU (L40/L40S/RTX 6000 Ada, or A40 for the dedicated-pod fallback)** — the sizing baseline the research sweep found for a 30B-class Qwen-VL model plus its (larger-than-text-only) image-token KV cache; A40 (~$0.44/hr) is the cited cheap anchor for the dedicated-pod path if serverless calibrates poorly.

## Design decisions

Explicit decisions made in response to review, so the reasoning isn't re-litigated later:

1. **Idle signal is `scan-state.json`'s `phase` field, not `running`.** `running` is true across the whole Scan including Tier 0/1 (scraping, CPU pHash/quality gate, SigLIP ranking) — none of which calls the vision backend. Using `phase !== "analyzing"` as idle means a dedicated pod isn't billed as "busy" for however long Tier 0/1 takes before the first Tier-2 call. Rationale: the watchdog's whole purpose is not paying for GPU time nothing is using; `running` alone measures the wrong thing.
2. **Watchdog mechanism is algo-corpus's real inline single-execution recheck, not a persistent `/tmp` counter file.** The real `runpod_ingest_watchdog.sh` does: read busy() once per invocation; if drained, `sleep 60`, recheck once, then act — entirely within one process, one systemd-timer firing. It does not need state to survive between firings. Rationale: a `/tmp` counter file surviving across separate cron/timer invocations and a reboot is a fragile dependency this plan doesn't need — the proven mechanism is simpler and already works in production for algo-corpus. Don't add the complexity back.
3. **Watchdog fails loud on a GraphQL error or curl failure, distinct from its normal no-op paths.** algo-corpus's own Python parsing collapses an auth error or malformed GraphQL response into the same "no pods found" result as a genuinely empty pod list. That's the exact failure this watchdog exists to prevent, wearing a harmless disguise. estate-scraper's version checks for a curl failure or a top-level `errors` key and sends a distinctly-worded ntfy alert plus a distinct log line, so a broken `RUNPOD_API_KEY` or an API outage is visible, not silent.
4. **Pod-name matching is exact-match or strict prefix (`estate-scraper-vision-*`), never a substring, and must be unit-tested.** A loose substring match on a shared RunPod account (same key, same account as algo-corpus's own dedicated pods) risks stopping — or failing to stop — the wrong project's GPU. The match logic must have a unit test asserting a mock pod list containing an algo-corpus-style name (e.g. `corpus-extraction-a1b2`) does *not* match `estate-scraper-vision-*`, and that an estate-scraper pod name does.
5. **A dedicated pod must never be provisioned before the watchdog exists and is verified working.** Ordering hazard, not just a feature: if calibration or production use ever provisions a dedicated pod ahead of the watchdog being deployed and its idle-detection/`podStop` path smoke-tested, that pod bills unattended the moment nothing is watching it — the precise incident this whole cutover exists to prevent. Serverless-first (Technology choices) sidesteps this during calibration; the watchdog ships and is verified before any dedicated pod is ever started.

### Rejected findings (left unchanged, on purpose)

- **No SQL `CHECK`/`NOT NULL` constraint on `vlm_model`.** This schema has no `CHECK` constraints anywhere, and Gemini-origin rows already rely on the same app-only invariant (`persist.ts` stamping `activeVlmModel()`). Adding a constraint here alone would be inconsistent over-engineering relative to the rest of the schema.
- **No path-traversal validation on `calibrate-runpod.ts`'s `--baseline`/`--candidate` CLI args.** This is a local operator-run dev tool, not a network-facing service — path traversal isn't a real threat model for a script the operator invokes against their own filesystem.
- **Pod IDs stay in ntfy alerts.** They're non-secret identifiers (not credentials, not customer data); redacting them isn't worth the engineering effort.

## Risk areas

- **Qwen3-VL reasoning-mode empty-response bug may reproduce on `runVisionManaged`.** Already hit and fixed once in this codebase for `runLocalGate` (see `LOCAL_GATE_MAX_TOKENS`'s comment in `vision.ts`): at a low token budget, a reasoning-tuned qwen3-vl model burns its budget on chain-of-thought and returns empty `content` on every call, which — combined with this codebase's fail-open-by-construction error handling — looks like silent success (findings simply never generated) rather than a visible failure. `runVisionManaged`/`VISION_SYSTEM_PROMPT` carry no `/no_think`-equivalent guard today, and calibration is the first time this path runs against a qwen3-vl-family model in production shape. Calibration must explicitly test for empty or truncated responses before trusting any other calibration figure — a false "0 findings" reading here would look like a real quality regression rather than the same bug recurring in a new call site. See Technology choices for the in-scope fix path if it reproduces.
- **Feasibility risk: the exact checkpoint may not load.** The research backing `worker-vllm` + Qwen3-VL-32B-Instruct was gathered against Qwen2.5-VL forum threads, not this specific checkpoint. Discovering a load/serve failure only after the watchdog and calibration machinery are fully built would waste that work; the feasibility spike (Technology choices) is ordered first specifically to catch this early and cheaply, with a named fallback path.
- **Serverless vision throughput is unproven, not just untested.** The research sweep independently found this flagged as an open problem in the vLLM community for Qwen2.5-VL, mirroring algo-corpus's own real ~6 chunks/min serverless failure on a text model. If it reproduces here, a Scan's Tier-2 pass could take far longer than the weekly schedule tolerates, discovered only if calibration actually measures wall-clock throughput and cold-start latency separately from steady-state per-image latency (not just backend agreement rate) — which is why requirement 7 insists on both.
- **GPU memory sizing for the vision KV cache.** Image tokens inflate vLLM's KV cache well beyond a text-only model of the same parameter count; a `--gpu-memory-utilization`/`--max-model-len` combination copied from a text deployment (e.g. algo-corpus's own 72B text pod config) could OOM mid-batch or silently truncate context on a vision model, degrading finding quality in a way that looks like a prompt regression rather than a sizing bug.
- **Cross-project pod collision on a shared RunPod account.** The watchdog's GraphQL query (`myself { pods { ... } }`) returns every pod on the account, including algo-corpus's own dedicated pods (same `RUNPOD_API_KEY` convention, same account). Mitigated by an exact/strict-prefix pod-name match with a unit test proving an algo-corpus-style name never matches (see Design decisions) — a wrong or loose match stops, or fails to stop, the wrong project's GPU.
- **Watchdog silent failure = the exact runaway-billing scenario it exists to prevent.** If the estate-scraper API process is down (so `scan-state.json` goes stale) or the RunPod API key expires/rotates without the watchdog's `.env` being updated, a naively-written watchdog degrades to "does nothing" with no signal — mirroring the precise failure this cutover is meant to close. Mitigated by the explicit fail-loud-on-GraphQL/read-error behavior in Design decisions and Integration points: a read failure or GraphQL error triggers a distinct ntfy alert, never a quiet no-op indistinguishable from "still busy."
- **An abandoned or failed calibration attempt can leave a dedicated pod billing indefinitely with nothing watching it** — the exact incident this feature exists to prevent, and no step in this plan currently deprovisions one. Mitigated two ways: (1) once the watchdog is deployed and verified (which must happen before any dedicated pod is provisioned — see Design decisions), idle-detection naturally stops an abandoned pod on its own; (2) if calibration is abandoned or fails *before* the watchdog exists, `CALIBRATION.md` must instruct an explicit manual `podTerminate` (or `podStop`) call as a teardown step — a dedicated pod is never left running "to come back to later" without either the watchdog or a manual stop covering it.
- **Backend agreement rate against a Gemini-shaped scorer may under- or over-state a genuine quality difference.** The existing `hasFindings()`/scoring heuristics (junk-line detection, verbose-prefix rejection, line-count/length caps) were tuned against Gemini's response style; Qwen-VL's plain-text phrasing conventions may differ enough to trip those heuristics independent of whether the underlying identification is actually as good — a calibration agreement-rate drop could reflect a scoring-heuristic mismatch rather than a real Tier-2 quality regression. Mitigated by the required 30-item disagreement spot-check (Integration points) before the calibration decision is recorded, not a vague "spot-check by eye" with no defined sample size.
