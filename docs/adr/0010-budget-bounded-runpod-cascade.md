# 0010 — Budget-bounded RunPod cascade, calibrated by a one-time reference pass

A weekly Scan produces ~6,000 images. Running the strong VLM on every surviving image via RunPod serverless costs ~$0.002–0.005/image ≈ $50–90/mo, 10–18× over the ~$5/mo target; running the full pass on the free home GPU works but takes ~23h and ties up the desktop (gaming/Plex). Both are rejected. Instead the pipeline is a tiered RunPod cascade where **the expensive tier is bounded by a dollar budget, not a score threshold**: Tier 0 (pHash dedup + quality gate, CPU, free) → Tier 1 (CLIP/SigLIP embeddings rank every survivor by "plausibly worth a closer look," ~$0.01–0.05/scan, batched, no generation) → Tier 2 (strong 32B VLM runs only on the top-K images, where `K = budget ÷ cost_per_image`). Allocation is **per-sale floor + remainder**: every sale clearing Tier 0 gets a guaranteed minimum (~4 images) to the 32B, then leftover budget is spent by global Tier-1 rank — so photo-dense sales can't starve the rest. The cheap tiers only do coarse **noise removal**; they are deliberately never asked to judge *value* (that's Tier 2's job), which keeps false-negatives — dropped good items — low. Endpoints scale to zero so cost is $0 outside the weekly Scan.

The budget is a dial the operator sets, and the cascade is calibrated against a **one-time reference pass**: a single money-no-object Scan sending all ~6,000 images through the 32B (~$22, frozen). Thereafter every cheap-tier choice (K, the per-sale floor, the CLIP prompts/exemplars) is tuned offline for free as `recall@K` against that frozen reference — yielding a "$X/mo captures Y% of full-pass findings" curve so frugality is provably calibrated, not guessed. The trade-off: the cheap ranker cannot distinguish a Stickley from an IKEA piece, accepted because it only needs to decide *which images deserve one expensive look*, and the reference pass plus the human gold set (expanded to include negatives/traps) measure how often that decision is wrong.
