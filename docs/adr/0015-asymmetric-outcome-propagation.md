# 0015 — Asymmetric Outcome propagation: bulk negatives, human-only positives

The taste ranker trains on Images, but the irreplaceable human signal — the [Outcome](0011-value-aware-identifier-comps-deferred.md) — is authored at the **Sale** level (good / meh / waste). Bridging that gap naively, by propagating a Sale's verdict to all its Images, poisons the corpus: a sale is "good" because of *one* item among dozens of junk photos, so stamping "good" on all of them feeds the ranker a flood of false positives and it learns that tube socks are desirable. The decision is therefore to propagate **asymmetrically**, because the two verdicts are not symmetric in what they prove.

A **"waste"** Outcome is a clean, complete signal: the user drove there and the whole sale was junk, so *every* Image in that Sale is a confirmed **negative**. These propagate in bulk — free, accurate, and exactly the "confirmed-junk exemplars" the Tier-1 ranker needs. A **"good"** Outcome proves only that *something* in the sale was worth it, not *what*, so it **never** auto-propagates to Images. Positive labels are instead attributed by a human tapping the specific Finding/Item that earned the visit, which writes an Item row with `source='human'`, `id_confidence='high'` — a clean `(embedding, confirmed-good)` pair. Item-level confirmation is optional and frictionless-when-offered, never required; the mandatory one-tap Sale Outcome remains the coarse signal and the source of bulk negatives. The trade-off accepted: positive training pairs accrue only as fast as the user bothers to confirm items, which is slower than auto-labeling — but auto-labeling positives is the poison path, and a smaller clean positive class beats a large contradictory one.
