# 0017 — SigLIP image embeddings over an OpenAI-compatible endpoint, computed as a post-scan pass from the thumbnail

[ADR 0013](0013-persist-embeddings-thumbnails-at-scan-time.md) decided *that* every analyzed image gets an embedding and a thumbnail, and [ADR 0016](0016-generator-provenance-frozen-embedding-model.md) decided the embedding model is *frozen and stamped* — but neither named the model, the serving protocol, or where in the pipeline the vector is produced. This ADR fixes those three so the irreplaceable embedding tier is concrete.

**Model: SigLIP** (`siglip-so400m-patch14-384`, 1152-dim), frozen per ADR 0016 and stamped as `embed_model`/`embed_dim` on every row. SigLIP's sigmoid contrastive objective outperforms CLIP on image–image and image–text retrieval at this scale, and retrieval is precisely the Tier-1 job: rank a new sale's photos against the user's confirmed-good corpus. The dimension is recorded so a cosine query can refuse to compare across a model change.

**Wire format: OpenAI-compatible `/embeddings`** (`{model, input: [data-uri]}` → `{data: [{embedding, index}]}`), configured by `EMBED_API_BASE` and pointed at the **broker batch port** in production so the shared GPU stays arbitrated (never raw Ollama — Ollama does not serve image embeddings anyway). The wire format is the stable contract; the serving stack behind it (Infinity, TEI, vLLM, a custom SigLIP server) can change without code changes. Unset `EMBED_API_BASE` disables the pass cleanly for dev. `EMBED_DIM` is a guard: a response of the wrong length is rejected rather than written, so a misconfigured endpoint can never silently poison the frozen column with another model's geometry.

**Production point: a post-scan pass that embeds from the persisted thumbnail, not inline in the vision stream.** Three reasons. (1) ADR 0016 already makes the thumbnail the canonical re-embed source, so embedding-from-thumbnail *is* the blessed path — and this same pass becomes the re-embed migration when the frozen model changes (null the column, re-run). (2) Batching many thumbnails per GPU request is far more efficient than one-at-a-time inline calls, and reading from local disk avoids re-fetching the soon-dead CDN URL. (3) It keeps 1152-float vectors out of the `analyzed_image` event stream, which is serialized to `scan-state.json` for the live dashboard. The pass is idempotent (fills only NULL embeddings, skips boilerplate) and resumable after a crash; because it runs after findings and image rows are committed, a failure there never loses the scan's primary output. The trade-off — embeddings lag the thumbnail by one pass rather than being computed in the same loop ADR 0013 imagined — is accepted: the Tier-1 ranker that would consume an inline embedding does not exist yet, and decoupling means a down embedding endpoint degrades to "thumbnails saved, vectors filled next run" instead of a failed scan.
