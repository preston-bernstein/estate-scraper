# 0011 — Value-aware identifier; comps deferred but not designed out

The tool's job is triage — get the user to the right driveway — not appraisal. It is an **Identifier** ("this sale has items matching your taste, worth a look"), not a Deal-Finder ("this Stickley is worth $2,000, priced at $80"). The latter requires external comps data (eBay sold, WorthPoint, auction results) with ongoing cost and ToS risk, and is deferred. But an identifier still needs a sense of what is *worth* recommending, sourced for free from three rungs: (1) the VLM's own latent value prior, surfaced by asking the existing top-K call to also emit a **Desirability** tier (high/med/low) — zero extra calls; (2) a curated maker/marque **lexicon** (Stickley, Henredon, Eames, Atari, …) as a deterministic value flag; (3) the `sale_outcomes` flywheel, which re-weights the generic prior toward the user's actual market over time and feeds the Tier-1 CLIP ranker as confirmed-good/confirmed-junk exemplars. To keep the door open to real valuation later, Findings are stored **normalized** (`maker`, `category`, `era`, `desirability`, `matched_lexicon`) rather than as a free-text blob, so a future comps lookup attaches as a join on `maker + category` without re-architecting. The trade-off: v1 cannot quote an actual price, accepted because the user attends in person with a phone and the hard problem the tool uniquely solves is choosing *which sales* are worth a Saturday.
