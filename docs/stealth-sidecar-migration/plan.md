# Plan: Stealth Sidecar Migration for HTML Scraping

## Approach
Rewrite `fetchText()` in `api/src/lib/http.ts` to drive the scraper-commons sidecar's `/v1` HTTP API instead of bare `fetch()`, while leaving `fetchBuffer()` and `politeDelay()` byte-for-byte unchanged. A new `api/src/lib/stealth-sidecar/` module cluster (`client.ts`, `errors.ts`, `session.ts`) ports the pattern already proven in fashion-monitor's `packages/core/src/platforms/stealth-sidecar/` — a thin 1:1 HTTP wrapper, two typed error classes (unreachable vs. sidecar-response), and session-lifecycle helpers — adapted to estate-scraper's simpler needs (no per-account profile persistence, one Scan-scoped shared context instead of a profile-keyed map). Because the sidecar serializes all browser operations on a single worker thread and the Scan/import loops are already strictly sequential, no concurrency-handling is needed anywhere in this change; the only new state is a lazily-created, Scan-scoped browser context that is explicitly torn down when the scraping batch finishes.

## Architecture
```
scraper/index.ts (scrapeWithinRadius)      import/index.ts (main)
        |  fetchText(url) x N                     |  fetchText(url) x N
        v                                          v
              api/src/lib/http.ts: fetchText()
                        |
                        v
        api/src/lib/stealth-sidecar/session.ts
   (shared context cache, per-call ephemeral page,
    stale-context retry, closeSidecarSession())
                        |
                        v
        api/src/lib/stealth-sidecar/client.ts
  (createContext / createPage / navigate / getContent /
              closePage / closeContext)
                        |  plain fetch(), STEALTH_SIDECAR_URL
                        v
        scraper-commons sidecar  (FastAPI, /v1, localhost:8000)
                        |  single worker thread, patchright
                        v
                 estatesales.net (via real stealth browser)
```
`fetchBuffer()` is untouched and keeps calling `fetch()` directly against CDN image hosts — it never touches the sidecar.

Session lifecycle (the load-bearing design decision): **one shared sidecar context per batch of `fetchText` calls, with a fresh page created and closed per call.**
- `scrapeWithinRadius()` (the Scan's entire scraping phase — 1 listing-page fetch + up to `maxSales` detail-page fetches, all sequential with `politeDelay()` between them) is the natural batch boundary. It already loops sequentially with no concurrency, matching the sidecar's single-worker constraint exactly as-is (NFR: Concurrency).
- On the first `fetchText` call in a process, `session.ts` lazily calls `createContext({})` once and caches the resulting `context_id` for the rest of the process's life (a promise-cached singleton, following fashion-monitor's `persistentContexts` pattern in `session.ts`, but a single global slot instead of a profile-keyed map since estate-scraper has exactly one caller-batch per process invocation and no per-account profiles to key on). The cached promise must self-heal on rejection: if `createContext` fails (e.g. the sidecar returns a transient 500), `session.ts` clears the cached slot via a `.catch()` on the cached promise itself so the next `fetchText` call retries `createContext` fresh, rather than replaying the same rejected promise for the rest of the batch — the exact bug class fashion-monitor's own `session.ts` hit and fixed. Because a newer promise may already have replaced the rejected one by the time the `.catch()` runs (e.g. after a stale-context-triggered recreate), the eviction must be guarded by an identity check — only clear the slot if it still holds the same promise that just rejected — mirroring fashion-monitor's `persistentContexts.delete(profilePath)` guard.
- Every `fetchText` call creates one page in that shared context, navigates it, reads its content, and closes the page in a `finally` (best-effort — a close failure is logged, not thrown, so it never masks the real result). Pages never accumulate: at most one is open at a time, closed before `fetchText` returns.
- The context itself is closed once, at the end of the batch, via `closeSidecarSession()`, called from a `try/finally` wrapped around `scrapeWithinRadius()`'s loop body and around `import/index.ts`'s `main()` loop body — not at the `fetchText`/`politeDelay` call sites themselves (those stay untouched per AC-6). `closeSidecarSession()` must catch and log — never throw — any error from its own close attempt, with no exceptions, exactly like the per-call page close above: because this call sits inside a `finally`, an exception thrown here would replace whatever exception was already propagating (per JS semantics), and the one case this matters most — the sidecar being genuinely unreachable — is exactly the case that already threw a clear `SidecarUnreachableError`. Letting the `finally` block's own connect failure clobber that would turn a clear, fail-fast error into a confusing "failed to close: ECONNREFUSED" and undermine the fail-fast clarity this whole feature exists for. This bounds context count to exactly 0 or 1 for the whole run (comfortably under the sidecar's `max_contexts=10` default), satisfying AC-11 by construction rather than by hoping cleanup runs.
- The combined navigate+content timeout budget (the per-call operation deadline used to distinguish a slow/hung page from a genuinely unreachable sidecar — see API/interface contract below) is owned by `session.ts`'s per-call page helper as a single shared deadline/`AbortController` spanning both the `navigate` and `getContent` calls for that page, not as two independent per-call timeouts inside `client.ts`. `client.ts`'s own per-request timeout (its inline fetch-with-timeout helper, see Technology choices) exists only to bound each individual HTTP round-trip to the sidecar and is not itself the operation-timeout budget; without a single deadline owned above the per-request level, two independent client.ts timeouts summed together could silently double the real worst-case latency.
- Why not a fresh context per call? Not a capacity concern — a context-per-call design would close each context immediately after use (symmetric to how pages already work), so it would never hold more than one slot at a time regardless of listing count, same as the shared-context design. The real reason is overhead and fingerprint consistency: context creation carries real browser-launch/context-setup cost, and a fresh context gets a freshly-randomized stealth fingerprint every time; sharing one context across a batch amortizes that creation cost across every listing/detail-page fetch in the same Scan and keeps a single, consistent fingerprint for the whole Scan rather than a different one per page. Why not one process-lifetime context with no explicit close? Both `scraper/index.ts` (`npm run scan`) and `import/index.ts` (`npm run import`) are one-shot CLI invocations, but relying on process-exit alone to clean up sidecar-side state does nothing — the sidecar is a separate long-lived process unaffected by the Node client dying, and Node's `beforeExit` hook does not fire on the explicit `process.exit(1)` used in both files' error paths, which is exactly the case (a failed Scan) where leaked contexts matter most. An explicit `try/finally` around the batch guarantees the close attempt fires on both the success and the fail-fast-throw paths.

## Data model
No data model changes.

## API / interface contract
No new sidecar endpoints (per constraint) — this feature only calls the existing `/v1` routes, exactly as documented in `scraper_commons/sidecar/schemas.py` and `app.py`:

| Call | Route | Request body | Response body | Notes |
|---|---|---|---|---|
| Create context | `POST /v1/contexts` | raw stealth-options dict, e.g. `{}` (NOT wrapped in an `"options"` key — verified directly against `app.py`'s `create_context(opts: dict[str, Any] = Body(default_factory=dict))`, which does not use `ContextCreateRequest`) | `{ context_id }` | Empty `{}` is enough — no login/profile persistence needed (constraint) |
| Create page | `POST /v1/contexts/{context_id}/pages` | — | `{ page_id }` | 404 `not_found` if `context_id` is stale |
| Navigate | `POST /v1/pages/{page_id}/navigate` | `{ url, timeout_ms? }` | `{ status, url }` — `status` is the **target site's** HTTP status from `page.goto()`, not the sidecar's own HTTP status | 404 `not_found` if `page_id` is stale; 422 `invalid_timeout` if `timeout_ms` is omitted-then-capped incorrectly (avoided by capping client-side) |
| Get content | `GET /v1/pages/{page_id}/content` | — | `{ content }` | No native per-call timeout on the sidecar side (FR-6) — the combined navigate+content deadline is enforced client-side, owned by `session.ts` (see Session lifecycle) |
| Close page | `DELETE /v1/pages/{page_id}` | — | 204 | Best-effort; 404 treated as already-closed, not an error |
| Close context | `DELETE /v1/contexts/{context_id}` | — | 204 | Best-effort; 404 treated as already-closed, not an error |

Sidecar error envelope (all non-2xx responses, per `errors.py`): `{"error": {"type": "<not_found|invalid_timeout|invalid_option|invalid_url|capacity_exceeded|operation_timeout|internal_error>", "message": "..."}}`. `SidecarResponseError`'s parsing of this envelope treats `errorType` as an opaque string and preserves it as-is rather than validating it against this fixed set — a future scraper-commons release adding a new `errorType` value must not cause estate-scraper's client to throw a client-side parse error; this forward-compatibility is cheap and cross-repo-safe to build in now.

`fetchText(url: string): Promise<string | null>` signature, export name, and null-on-failure contract are unchanged (FR-1, FR-11). Its new failure modes, all satisfying FR-3/FR-4/FR-5/FR-6/FR-10/AC-2/AC-3/AC-4/AC-5 — requirements.md's FR-4/FR-6/AC-3/AC-5 draw an explicit line between connect-level unreachability and per-call operation timeouts, and this plan follows that line exactly:
- Target page's `NavigateResponse.status` is non-2xx → log URL + status, return `null` (same as today's `!response.ok`).
- Sidecar unreachable at the CONNECT level — the fetch to the sidecar's own HTTP endpoint itself rejects (ECONNREFUSED/ECONNRESET/ENOTFOUND/DNS failure), on any of the six `/v1` calls — → throw `SidecarUnreachableError` (new, exported from `stealth-sidecar/errors.ts`). Never falls back to `fetch()`. This is the only case that counts as "sidecar unreachable."
- Per-call operation timeout — navigate+content together exceeding the configured budget (20s) while the sidecar itself is reachable, just slow or hung on this one target page — → treated exactly like a non-2xx target-site status: log the URL, return `null`, let the caller continue. This is deliberately NOT a `SidecarUnreachableError`: a single slow/hung estatesales.net page is a per-URL problem, not evidence the sidecar itself is down, and `scraper/index.ts`'s fetch loop has no try/catch around individual `fetchText` calls — treating a timeout as fail-fast would abort the entire remaining Scan over one bad page, a real regression from today's graceful-skip-and-continue behavior (FR-4/FR-6, AC-3/AC-5).
- Sidecar reports 404 `not_found` for the cached context/page → recreate context + page once, retry navigate+content once; surface that retry's own outcome (null or throw, per the rules above) — the stale-ID 404 itself is never surfaced to the caller (FR-5, AC-4).
- Any other non-2xx sidecar-level error (422/429/500/504 from context/page create, navigate, or content) → throw `SidecarResponseError` (new, exported from `stealth-sidecar/errors.ts`) — these represent the sidecar failing to do its job, not the target site returning a bad status, so they get the same fail-fast treatment as "unreachable" rather than being silently swallowed into `null`.

`politeDelay()` — no interface change (FR-8, AC-7). `fetchBuffer()` — no interface change (FR-9, AC-8).

## Integration points
- `api/src/lib/http.ts` — replace `fetchText`'s body with a call into `./stealth-sidecar/session.ts`; catch `SidecarUnreachableError`/`SidecarResponseError` from the new fail-fast paths and let them propagate (do not catch-and-null them); leave `fetchBuffer` and `politeDelay` untouched; drop the `FETCH_HEADERS` import for `fetchText` (still imported for `fetchBuffer`, unaffected).
- `api/src/lib/stealth-sidecar/client.ts` (new) — 1:1 wrapper around the six `/v1` routes above, ported from fashion-monitor's `client.ts` pattern; owns request timeouts and JSON (de)serialization; no scraper-specific logic. Fashion-monitor's `client.ts` imports a `fetchWithTimeout` helper from its own `../../lib/http.js` — estate-scraper's `api/src/lib/http.ts` has no such export (only `fetchText`/`politeDelay`/`fetchBuffer`, each inlining `AbortSignal.timeout` directly), and importing one from `http.js` anyway would create a circular import (`http.ts` → `stealth-sidecar/session.ts` → `stealth-sidecar/client.ts` → `http.ts`, since `http.ts` is the file importing FROM the sidecar module in the first place). `client.ts` must therefore implement its own small inline fetch-with-timeout helper (a local `AbortController` + `setTimeout` wrapper, or equivalent) — not import one from `http.ts` or anywhere under `lib/`. Two additional hardening details belong here: URL-encode `context_id`/`page_id` when interpolating them into request paths (defensive best practice, even though these IDs are sidecar-generated, not user input); and validate that `STEALTH_SIDECAR_URL` resolves to a loopback or private-network address, operationalizing the NFR that already assumes localhost/private-network-only reachability with an actual runtime check rather than leaving it an unenforced assumption.
- `api/src/lib/stealth-sidecar/errors.ts` (new) — `SidecarError` base, `SidecarUnreachableError`, `SidecarResponseError` (with `status` + sidecar `errorType`, the latter parsed as an opaque string so an unrecognized future value from scraper-commons passes through rather than throwing a client-side parse error), ported from fashion-monitor's `errors.ts`.
- `api/src/lib/stealth-sidecar/session.ts` (new) — shared-context cache (`getOrCreateSharedContext`, evicting its cached promise on rejection with an identity-check guard so a transient `createContext` failure self-heals instead of permanently poisoning the batch), `withEphemeralPage`-per-call helper (owns the combined navigate+content operation-timeout deadline as a single shared `AbortController`/deadline per call, not two independent client.ts-level timeouts), stale-context invalidation + one retry, `closeSidecarSession()` for explicit teardown (must catch and log, never throw, any error from its own close attempt).
- `api/src/lib/scraping.ts` — no code change; `FETCH_HEADERS` stays exactly as-is (still consumed by `fetchBuffer`). Add a one-line comment noting it no longer applies to `fetchText`'s HTML fetches now that a real stealth browser sets its own fingerprint/headers, so the two hand-rolled fields don't silently look stale to a future reader.
- `api/src/scraper/index.ts` — wrap `scrapeWithinRadius`'s body in `try { ... } finally { await closeSidecarSession(); }`; no change to the `fetchText(...)`/`politeDelay()` call sites themselves (FR-8, AC-6).
- `api/src/import/index.ts` — wrap the ENTIRE `main()` function body (not per-loop-iteration) in the same `try { ... } finally { await closeSidecarSession(); }` pattern; no change to `ensureSaleRecord`'s `fetchText(entry.url)` call site.
- `api/src/scan/index.ts` — the actual production CLI entrypoint (`npm run scan` → `tsx src/scan/index.ts`; `api/src/lib/scan-runner.ts` spawns the compiled `scan/index.js` as an isolated child process for a manually-triggered scan, confirming the one-process-per-batch assumption this whole design leans on holds in production, not just in the `npm run scan`/`npm run import` CLI cases already covered above). Its `main()` already wraps the whole scrape/analyze pipeline in a single `try/catch`: on any thrown error it flattens the error down to `error instanceof Error ? error.message : "Unknown scan error"`, pushes that string via `writer.pushEvent({ type: "error", msg: message })` and `writer.finish(message, true)`, logs it with `console.error(message)`, and sets `process.exitCode = 1` — it does not call `process.exit()` in this handler (the separate module-level `main().catch(...)` is a belt-and-suspenders handler for a rejection escaping `main()` itself, and that one does log the raw error and call `process.exit(1)`). Because the in-`main()` path keeps only `.message`, richer typed fields like `SidecarResponseError.status`/`errorType` do not survive to reach `ScanStateWriter` or the console. `SidecarUnreachableError`/`SidecarResponseError`'s `.message` text itself must therefore contain an identifiable marker (e.g. the literal word "sidecar") so a sidecar-attributable failure stays recognizable in the persisted scan state and console log even after this flattening — no other code change to `scan/index.ts` is required. A successful `tsc`/build step remains a precondition for `scan-runner.ts`'s spawn of the compiled `scan/index.js` to pick up this migration's code in production; a `npm run dev` (tsx-on-src) check alone does not exercise that production path.
- `api/src/lib/__tests__/http.test.ts` (new) — unit tests for `fetchText`'s new branches (2xx/non-2xx navigate status, connect-level unreachable, stale-context retry, per-call operation timeout treated as a graceful `null`), mocking `fetch` with `vi.stubGlobal("fetch", ...)` — no new test-mocking dependency.
- `api/src/lib/stealth-sidecar/__tests__/*.test.ts` (new) — unit tests for `client.ts` (request shaping, error-envelope parsing) and `session.ts` (shared-context caching, rejected-promise cache eviction, stale-id invalidation, `closeSidecarSession` best-effort/never-throw behavior). `session.ts`'s module-level singleton context cache needs an exported test-only reset/drain mechanism (e.g. a `__resetForTests()` or equivalent) so these tests — and any other test file that imports `session.ts` — don't leak cached context state into each other; mirror fashion-monitor's own `session.test.ts` pattern of draining the cache before resetting mocks in `beforeEach`.
- `api/src/scraper/__tests__/index.test.ts` (new, no such file exists today — `scraper/__tests__/` currently only has `parse.test.ts`) — the AC-9/AC-10 Scan-level checks: a full `scrapeWithinRadius()` run against a mocked sidecar producing the same `ScrapedSale[]` shape as the pre-migration fixture-based test would have, and a second test asserting `scrapeWithinRadius()` rejects (rather than returning a truncated array) when the mocked sidecar is unreachable at the connect level — plus a third test covering the graceful-degrade path, where a mocked slow/hung navigate or content call resolves to a logged `null` for that one URL while the Scan's loop continues to the next.
- `api/.env.example` — document `STEALTH_SIDECAR_URL` (default `http://127.0.0.1:8000`), matching the reachability constraint (local/private-network only).

## Technology choices
- Port (not import) fashion-monitor's `client.ts`/`session.ts`/`errors.ts` pattern into estate-scraper's own `api/src/lib/stealth-sidecar/` — required by the constraint that this feature not depend on fashion-monitor's package or build a cross-repo shared client.
- No new npm dependency: the sidecar client is plain `fetch()` + `AbortController`, with `client.ts` implementing its own small inline timeout wrapper the same way `http.ts`'s `fetchText`/`fetchBuffer` already inline `AbortSignal.timeout` — not by importing a `fetchWithTimeout` helper from `http.ts` or fashion-monitor's `lib/http.ts` (see Integration points for why that specific literal port doesn't work here); no HTTP client library is introduced.
- No new test-mocking library: `vitest`'s built-in `vi.stubGlobal` covers mocking global `fetch` for both the sidecar-client tests and the Scan-level integration-style tests, consistent with `api/package.json`'s existing devDependency set (vitest + stryker only).
- Deliberate deviation from fashion-monitor's reference: no blind connect-level retry-with-delay in `client.ts`. Fashion-monitor's `fetchWithRetry` retries once after 200ms on any connect-level failure; estate-scraper's requirements explicitly scope out any retry/backoff beyond the single FR-5 stale-context-ID retry ("no multi-attempt exponential backoff... unless the current `fetch()` implementation already had equivalent behavior (it does not)"). A sidecar-unreachable condition here fails on the first attempt.

## Risk areas
- **Session-lifecycle correctness is the crux of this migration.** Getting the shared-context-plus-per-call-page pattern right (especially the stale-context detection-and-recreate path and guaranteeing `closeSidecarSession()` fires on every exit from `scrapeWithinRadius`/`import`'s `main`, including thrown errors) is more state-machine-shaped than the rest of this change combined, and is exactly the surface FR-5/FR-7/AC-4/AC-11 are testing.
- **`NavigateResponse.status` semantics.** It is the target site's HTTP status from `page.goto()`, not the sidecar's own HTTP status — confirmed by reading `app.py` directly rather than assuming from the schema alone (fashion-monitor's own `client.ts` flagged the analogous `content` field as an unverified assumption; this plan closes that gap for `navigate`, but a mismatch here would silently break FR-3/AC-2's non-2xx-status handling).
- **Timeout budget is a judgment call, not a measured one — but a lower-stakes one now.** 20s (vs. today's 15s) is chosen based on the sidecar's default `op_timeout_ms=30s` and the general expectation that a real browser round-trip is slower than a bare `fetch()`, but no real-world timing data against a live sidecar backs this number yet. Because a per-call operation timeout is now a graceful-degrade path (log + `null`, same as a non-2xx status) rather than a fail-fast `SidecarUnreachableError`, a too-tight 20s budget no longer risks aborting an entire Scan — it only causes more individual pages to be skipped, a real but much smaller-blast-radius failure mode than before. Still worth tuning once tested against an actual local sidecar instance per the constraint that the feature "must be buildable and testable against a locally-run sidecar instance now," but it is no longer a scan-aborting risk.
- **AC-9/AC-10's "Scan or its unit-test equivalent" are satisfied here via mocked-`fetch` tests, not a real running sidecar in CI.** This is explicitly permitted by the acceptance criteria's own wording, but it means a genuine end-to-end pass against a live local sidecar process is still a manual verification step before this is considered done, not something the automated test suite proves on its own. The mocked test suite must cover the timeout-as-graceful-degrade path explicitly (a mocked slow/hung navigate or content call resolving to a logged `null`, with the Scan's loop continuing to the next URL) in addition to the connect-level `SidecarUnreachableError` path, since these are now two distinct, separately-tested branches rather than one merged fail-fast branch.
- **The sidecar being unreachable in production is an accepted, tracked gap** (per Constraints) — this plan's fail-fast behavior (FR-4) means a production Scan will hard-fail every week until the sidecar deployment gap closes. That is the intended, correct behavior per the requirements, but it is worth flagging loudly since it turns a currently-degrading feature into a currently-broken one until the separate home-infra/scraper-commons deployment work lands.
- **Interactive Ctrl-C (SIGINT) cancellation mid-run is not covered by the `try/finally` cleanup.** Node's default SIGINT handling can terminate the process before an in-flight network call's `finally` completes, so `closeSidecarSession()` may never run on a manual Ctrl-C. This is an accepted gap for this feature (no SIGINT handler is being added); the sidecar's own idle-context reaper is the eventual fallback cleanup mechanism for any context leaked this way.
- **If the sidecar is ever deployed as a single shared instance serving multiple repos** (fashion-monitor, estate-scraper, algo-factory, per the broader rollout), estate-scraper's context/capacity usage becomes a multi-tenant capacity question across those repos — out of scope for this feature to solve; a home-infra/scraper-commons deployment-time concern, not something this plan's design needs to account for.
- **`session.ts`'s module-level singleton context cache needs an exported test-only reset/drain mechanism** so unit tests in different files don't leak cached state into each other — mirror fashion-monitor's own `session.test.ts` pattern (drain before resetting mocks in `beforeEach`). Noted here as a risk as well as in Integration points since skipping it is the kind of gap that only surfaces later as flaky, order-dependent test failures.
