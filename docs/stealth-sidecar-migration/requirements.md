# Requirements: Stealth Sidecar Migration for HTML Scraping

## Problem statement
`api/src/lib/http.ts` fetches estatesales.net HTML (listing + detail pages) with plain `fetch()`, which presents no browser fingerprint, executes no JS, and is trivially distinguishable from a real browser by anti-bot defenses. The weekly Friday-1am Scan (`api/src/scraper/index.ts`) depends on this fetch succeeding across every listing and detail page in the configured radius; if estatesales.net starts blocking or challenging plain `fetch()` requests, the Scan silently degrades (partial results) or fails outright, and because a Scan only accumulates and never overwrites, a bad Scan is not obviously distinguishable from "there were just fewer sales this week." The scraper-commons stealth sidecar (a local FastAPI service driving a real stealth browser session) already exists and has been proven against two other real scrapers (fashion-monitor's Depop/Poshmark migration). This feature moves estate-scraper's HTML fetch path onto that sidecar so listing/detail scraping happens over a real browser session instead of a bare HTTP client, while leaving the separate image-binary download path alone.

## Users / stakeholders
- Estate-scraper's weekly Scan job (`api/src/scraper/index.ts`) — the primary caller of the migrated `fetchText`.
- Estate-scraper's production CLI entrypoint (`api/src/scan/index.ts`, run via `npm run scan` → `tsx src/scan/index.ts`) — the actual production entrypoint that invokes `scrapeWithinRadius()`; its own top-level error handling must be able to surface a sidecar-origin failure distinctly (see AC-12).
- The one-off import path (`api/src/import/index.ts`) — secondary caller of `fetchText`.
- Preston (operator) — depends on Scan reliability and on clear failure signals when the sidecar is unavailable, since silent fallback would corrupt the "accumulate, never overwrite" data model with partial/degraded data that looks legitimate.
- The vision pipeline (`api/src/vision/oracle.ts`, `api/src/vision/index.ts`) — consumer of `fetchBuffer`; affected only insofar as this feature must confirm `fetchBuffer` is explicitly out of scope, not silently left behind.
- scraper-commons/home-infra maintainers — own the sidecar service itself and the still-open production-deployment gap; this feature is a client of their contract, not an owner of it.

## Functional requirements
1. The system shall replace the `fetch()`-based implementation of `fetchText(url: string): Promise<string | null>` in `api/src/lib/http.ts` with an implementation that retrieves the URL's HTML via a stealth browser session against the scraper-commons sidecar's `/v1` HTTP API, preserving the existing function signature and null-on-failure return contract so `api/src/scraper/index.ts` and `api/src/import/index.ts` require no call-site changes.
2. The system shall, for each `fetchText` call, create or reuse a sidecar page (`POST /v1/contexts` then `POST /v1/contexts/{context_id}/pages` as needed), navigate it to the target URL via `POST /v1/pages/{page_id}/navigate`, and retrieve the rendered HTML via `GET /v1/pages/{page_id}/content`. The navigate response's `status` field reflects the TARGET SITE's HTTP status from the browser's own page load (e.g., estatesales.net returning 404 or 503), not the sidecar service's own HTTP response code for the `/v1/pages/{page_id}/navigate` call itself; the two are distinct and must not be conflated (see FR-3, FR-5).
3. The system shall treat a non-2xx `status` in the navigate response — i.e., the target site's own HTTP status as reported by the browser's page load, not the sidecar's HTTP response code (see FR-2) — the same way the current implementation treats a non-ok `fetch` response: log the URL and status, and return `null` rather than throwing.
4. The system shall treat a sidecar-unreachable condition — a CONNECT-level failure to the sidecar's own `/v1` HTTP endpoint (connection refused, connection reset, DNS failure/ENOTFOUND, or equivalent, on any of the six `/v1` calls: create context, create page, navigate, get content, and their `DELETE` variants) where the client cannot reach the sidecar's HTTP endpoint at all — as a distinct, fail-fast error path: `fetchText` shall raise (not return `null`, not fall back to `fetch()`) so a Scan run aborts visibly instead of silently proceeding with zero/partial pages, matching fashion-monitor's equivalent-migration behavior. A per-call operation TIMEOUT on an otherwise-reachable sidecar (navigate + content retrieval exceeding the budget in FR-7) is explicitly a different failure class from sidecar-unreachable and is NOT covered by this requirement; see FR-7 for its treatment.
5. The system shall treat any non-2xx response from the sidecar's OWN operations — context creation (`POST /v1/contexts`), page creation (`POST /v1/contexts/{context_id}/pages`), content retrieval (`GET /v1/pages/{page_id}/content`), or a sidecar-level error on the navigate call itself — as distinct from the target site's page-load status carried in the navigate response body (FR-2, FR-3). Such a sidecar-side operation error shall raise a distinguishable error and abort the current `fetchText` call, following the same fail-fast treatment as sidecar-unreachable (FR-4), rather than being silently swallowed into a `null` return; these responses represent the sidecar failing to perform its own operation, not the target site returning a bad page.
6. The system shall treat a sidecar-returned 404 for a context or page ID (e.g., after idle-reap or sidecar restart) by creating a fresh context/page and retrying navigation exactly once before surfacing failure for that URL, since the sidecar's registry is in-memory/non-persistent and IDs can go stale between calls. The retry's own outcome follows the same rules as any other `fetchText` call: a connect-level failure on the retry raises (FR-4), and a timeout or target-site non-2xx status on the retry returns `null` (FR-3, FR-7); the original stale-ID 404 itself is never the caller-visible error either way.
7. The system shall bound each `fetchText` call's navigate + content retrieval (combined) with an overall operation timeout whose default budget is 15 seconds, matching the current `AbortSignal.timeout(15_000)` behavior. The implementation may adopt a wider, justified ceiling if 15 seconds proves too tight for real-world sidecar round trips, provided the new value is explicitly stated and documented (per the Non-functional Requirements' latency provision) rather than left open-ended; this requirement does not fix a specific new number. A call that exceeds this budget is a per-call operation TIMEOUT, not a sidecar-unreachable condition (FR-4): it shall be treated the same as a target-site non-2xx status (FR-3) — logged and returned as `null`, allowing the Scan/import loop to continue to the next listing — not raised.
8. The system shall bound sidecar resource usage to a concrete, checkable limit: at most one shared sidecar context open at a time per batch (a Scan run or an import run), zero pages left open once any single `fetchText` call returns, and zero contexts left open once the batch completes. Any close/cleanup attempt (context or page, via `DELETE /v1/contexts/{id}` or `DELETE /v1/pages/{id}`) shall be best-effort and must never throw an exception that masks or replaces an already-in-flight fail-fast error from FR-4.
9. The system shall preserve `politeDelay()` unchanged and continue to invoke it between `fetchText` calls in `api/src/scraper/index.ts` exactly as today, since the sidecar migration governs how a page is fetched, not the pacing between fetches.
10. The system shall leave `fetchBuffer(url: string): Promise<Buffer | null>` in `api/src/lib/http.ts` on its current `fetch()`-based implementation, unchanged, because image binaries are served from a CDN/media host separate from estatesales.net's anti-bot-protected HTML pages and are out of scope for this migration (see Out of scope).
11. The system shall surface a clear, distinguishable error (not a generic thrown `Error` indistinguishable from a parse or network failure) when the sidecar is unreachable, so operators/logs can tell "sidecar down" apart from "estatesales.net returned an error" or "HTML didn't parse."
12. The system shall not change the return type, exported name, or call signature of `fetchText`, so existing callers (`api/src/scraper/index.ts`, `api/src/import/index.ts`) continue to compile and run without modification beyond what's needed to handle the new fail-fast error path.

## Non-functional requirements
- Per-request latency: a single `fetchText` call (navigate + content) shall complete within the same outer bound as today (15s), acknowledging real browser navigation is typically slower than a bare `fetch()` — if 15s proves too tight for real-world sidecar round trips, the implementation may widen the ceiling but must state and justify the new value; it must not be silently open-ended.
- Concurrency: the sidecar's worker thread serializes one browser operation at a time across all contexts/pages, so the migrated `fetchText` must not assume or rely on concurrent navigation/content calls succeeding in parallel — the Scan's existing sequential per-listing loop (with `politeDelay` between calls) already matches this constraint and shall not be changed to introduce concurrent scraping.
- Reachability: the sidecar is local-loopback-only with no auth/TLS; the client shall assume localhost/private-network reachability only and shall not attempt to reach a remote or public sidecar address.
- Data integrity: because a Scan accumulates and never overwrites prior data, a partially-failed Scan (due to sidecar unavailability) must not be indistinguishable from a legitimately-thin week — this is satisfied by the fail-fast requirement (FR-4) causing the Scan run to abort/error visibly rather than complete with silently-missing sales.

## Constraints
- Must integrate with the scraper-commons sidecar's existing `/v1` API contract exactly as built (`/v1/contexts`, `/v1/contexts/{id}/pages`, `/v1/pages/{id}/navigate`, `/v1/pages/{id}/content`, `DELETE` variants, `/v1/health`) — no new sidecar endpoints may be requested or assumed as part of this feature.
- The sidecar is not yet deployed as a reachable service on the production host; this is a known, tracked, out-of-scope gap owned by home-infra/scraper-commons. This feature must be buildable and testable against a locally-run sidecar instance now, and must fail fast (per FR-4) rather than degrade gracefully when the sidecar is absent in an environment (including production, until the deployment gap closes).
- `FETCH_HEADERS` (currently imported from `./scraping.js` and passed to `fetch()`) is estatesales.net-specific request-header tuning; its relevance/mechanism under a stealth browser session (which sets its own headers/fingerprint) must be reconciled during implementation, not assumed to carry over unchanged.
- fashion-monitor's `packages/core/src/platforms/stealth-sidecar/` client (HTTP wrapper, session helpers, typed error classes for unreachable vs 4xx/5xx) is available as a reference architecture but is a different codebase/package; this feature does not require importing or depending on that package directly — estate-scraper may port the pattern into its own `api/src/lib/` module.
- `fetchBuffer` and its callers (`api/src/vision/oracle.ts`, `api/src/vision/index.ts`) must continue to function unmodified; this feature must not introduce a regression in image-download behavior.
- No persistent per-account browser profile/login state is needed (estatesales.net requires no login), unlike fashion-monitor's Poshmark case — the implementation should not build session/profile-persistence machinery that this scraper has no use for.

## Out of scope
- Migrating `fetchBuffer` (image binary downloads) onto the sidecar. Image URLs are served from a CDN/media host, not estatesales.net's protected HTML pages, so there is no anti-bot rationale for routing binary downloads through a stealth browser session; `fetchBuffer` stays on `fetch()`.
- Deploying the sidecar as a reachable production service (container, process supervision, network exposure). That is a separate home-infra/scraper-commons tracked follow-up.
- Any change to the Scan's scheduling (Friday 1am), radius logic, geocoding, distance filtering, or the "never overwrite, only accumulate" data model.
- Any change to the vision cost-cascade pipeline itself.
- Building a general-purpose reusable sidecar-client package shared across repos (e.g., publishing estate-scraper's client back into `scraper-commons` or depending on fashion-monitor's package). Cross-repo sharing, if wanted later, is a separate decision.
- Retry/backoff policy beyond the single stale-ID retry in FR-6 (e.g., no multi-attempt exponential backoff on navigation failures) unless the current `fetch()` implementation already had equivalent behavior (it does not — it fails once and returns `null`).
- Screenshot capture (`GET /v1/pages/{page_id}/screenshot`) — not needed for HTML scraping and not used by this feature.
- Sidecar health-check-driven behavior (e.g., pre-flight `GET /v1/health` polling before every Scan) beyond what's needed to satisfy the fail-fast requirement; a full health-monitoring integration is not part of this feature.

## Acceptance criteria
1. `fetchText(url)` returns the target page's HTML string, sourced from the sidecar's `/v1/pages/{page_id}/content` response, for a URL the sidecar can successfully navigate to.
2. `fetchText(url)` returns `null` (does not throw) when the sidecar's navigate response reports a non-2xx status, and logs the URL and status.
3. `fetchText(url)` raises a distinguishable error (not `null`) when a CONNECT-level failure prevents reaching the sidecar's HTTP endpoint at all (connection refused, connection reset, DNS failure, or equivalent, on any `/v1` call) — sidecar-unreachable, per FR-4 — and does not fall back to a `fetch()`-based request.
4. Given a sidecar 404 on a stale context/page ID, `fetchText(url)` transparently creates a fresh context/page, retries navigation once, and either succeeds or surfaces the retry's own failure per criteria 2/3 above — it does not surface the stale-ID 404 itself as the caller-visible error. The retry's own outcome follows the same rules as any other `fetchText` call: a connect-level failure on the retry raises (per criterion 3), and a timeout or target-site non-2xx status on the retry returns `null` (per criteria 2/5) — the stale-ID 404 is never surfaced either way.
5. A `fetchText` call whose navigate + content retrieval exceeds the configured overall timeout (FR-7) returns `null` — logged the same as a target-site non-2xx status (criterion 2) — rather than hanging indefinitely or raising. A per-call timeout on an otherwise-reachable sidecar is never treated as sidecar-unreachable (criterion 3); those are distinct failure classes and this criterion is definitive, not left to implementation discretion.
6. `api/src/scraper/index.ts` and `api/src/import/index.ts` require no changes to their `fetchText`/`politeDelay` call sites' signatures or return-value handling beyond what's needed to handle the new fail-fast error path from criterion 3.
7. `politeDelay()` remains byte-for-byte unchanged in `api/src/lib/http.ts` and continues to be called between `fetchText` calls in the Scan loop exactly as before.
8. `fetchBuffer(url)` in `api/src/lib/http.ts` remains on its current `fetch()`-based implementation, unchanged, and `api/src/vision/oracle.ts` / `api/src/vision/index.ts` continue to call it successfully with no code changes required.
9. Running a Scan (or its unit-test equivalent) against a locally-running sidecar instance produces the same shape of scraped-sale data (`ScrapedSale[]`) as the pre-migration `fetch()`-based path did against the same fixture HTML.
10. Running a Scan (or its unit-test equivalent) with no sidecar running produces a visible, immediate failure (thrown error, non-zero exit, or equivalent) rather than a Scan that completes with zero or partial sales silently recorded as if legitimate.
11. Within any batch (a Scan run or an import run): at most one shared sidecar context is open at a time, zero pages remain open once any single `fetchText` call returns, and zero contexts remain open once the batch completes — verifiable by sidecar-side context/page counts returning to zero after the run. Any close/cleanup attempt (context or page) is best-effort and never throws an exception that masks or replaces an already-in-flight fail-fast error from criterion 3.
12. When `scrapeWithinRadius()` raises a sidecar-unreachable (FR-4) or sidecar-operation error (FR-5), `api/src/scan/index.ts`'s own top-level error handling (its catch/exit-code logic) surfaces a message that is clearly identifiable as a sidecar-origin failure — not indistinguishable from a generic scrape error — verified directly by inspecting the actual error text/exit behavior produced at that entrypoint, not assumed to happen automatically because the error originates deeper in the call stack.
