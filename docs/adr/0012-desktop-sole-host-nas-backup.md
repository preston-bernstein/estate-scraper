# 0012 — Desktop is the sole host; NAS for backup and bulk; scans on a systemd timer

The scheduling split the system across two machines: `deploy-remote.sh` ships the API/UI to the Linux desktop as a systemd `--user` service under a dedicated `estate-scraper` service user, while a macOS LaunchAgent (`com.estate-scraper.scan.plist`) ran the weekly scan on the laptop. Because the scan wrote to the laptop's `api/data/estate-scraper.db` and the API served the desktop's copy, there were effectively two divergent databases — the scan filled one, the dashboard read the other. The decision is to make the **always-on Linux desktop the single host** for everything: API, UI, the scheduled scan, and the one canonical SQLite database (local disk, WAL mode, under the service user's home — never on a network mount, which corrupts SQLite). The scan moves to a **systemd timer** on the desktop, and the macOS LaunchAgent is retired; the laptop becomes a dev-only machine whose local DB is throwaway.

The desktop wins on every axis that matters here: it is always on (a laptop sleeps, travels, and closes mid-scan), it holds the GPU so the free Tier-1 embedding pass runs locally with no network hop for ~6k images, and it already runs the service under the service-user convention. The **NAS** (lots of room, RAID + snapshots) is the durable tier — it holds thumbnails as plain files and receives a post-scan `sqlite3 .backup` of the database, but never hosts the live write path. **RunPod** remains the cloud Tier-2 only. Net result: one live host on local disk, a backup/bulk tier on the NAS, and the cloud reserved for the expensive model — and the two-database bug is gone. The trade-off is that the system is tied to the desktop being up and reachable for scans; accepted because the scan is a scheduled weekly batch, not an interactive service, and the desktop is the only always-on machine with the GPU.
