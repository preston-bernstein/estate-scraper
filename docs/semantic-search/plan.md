# Plan: Semantic Search

## Approach
Phase 1 adds a checked-in synonym/category thesaurus (`api/src/lib/thesaurus.ts`, sibling to `lexicon.ts`) that expands query terms before the existing lexical LIKE match in `searchSales`, plus a `finding_items.category` join so a query like "couch" also pulls every `seating` row regardless of description wording — zero API calls, fully deterministic. `thesaurus.ts` imports and reuses the existing `Category` type from `api/src/lib/items.ts` (the real 15-value enum) rather than inventing a parallel `FindingCategory` type — so Phase 1 is **not** fully isolated to `thesaurus.ts` + `discover.ts`; it has one intentional cross-module read dependency on `items.ts`'s type, acknowledged here rather than pretending otherwise. Phase 2 wraps Phase 1 with a new orchestrator that embeds the query into the frozen SigLIP space, ranks Findings by cosine similarity to their (already-persisted) image embedding, and fuses that ranking with the Phase 1 result via Reciprocal Rank Fusion (RRF) — falling back to Phase 1 alone whenever the embedding endpoint is unset, slow, erroring, disabled via kill switch, or returns the wrong dimension. `searchSalesHybrid` fetches `upcomingSales`/`upcomingSaleIds` once and passes them into both `searchSales` and the semantic ranker as an optional parameter, instead of each side independently re-querying — a pure refactor, not accepted debt. The two phases otherwise touch disjoint files: Phase 1 owns `lib/thesaurus.ts` + the lexical half of `services/discover.ts`; Phase 2 adds new files and only swaps one function call in the route, so they can be built in parallel worktrees and merged independently.

## Architecture

```
Phase 1 (always runs, no I/O beyond SQLite)
  searchSales(query, upcomingSales?)
    ├─ expandQuery(terms)              [lib/thesaurus.ts]
    │    → literalTerms, expandedTerms, categories: Category[]   (Category imported from lib/items.ts)
    ├─ LIKE match over findings.description using expandedTerms
    │    (each expanded term has %, _, \ escaped before the pattern is built — an unescaped "%" would match everything)
    ├─ finding_items.category IN (categories) → extra finding ids
    ├─ sale title/city/state/address substring match  (unchanged)
    └─ → RankedSale[]   (unchanged shape)

Phase 2 (additive orchestrator; only reachable via route swap; gated by SEMANTIC_SEARCH_ENABLED)
  routes/discover.ts  GET /search
    → searchSalesHybrid(query)          [services/semanticSearch.ts]
         ├─ if !SEMANTIC_SEARCH_ENABLED || !embeddingEnabled() → log("lexical-fallback:disabled") → return lexical
         ├─ fetch upcomingSales/upcomingSaleIds ONCE, shared with both branches below
         ├─ lexical = searchSales(query, upcomingSales)         (Phase 1, always computed — the fallback)
         ├─ vec = getCachedQueryEmbedding(query)  [lib/queryEmbedCache.ts → lib/embed.ts embedQueryText]
         │    (query length gated 2–200 chars; true-LRU cache capped at 512 entries; short timeout;
         │     EMBED_DIM is mandatory/fail-fast for this path; any failure → null)
         ├─ if vec === null → log("lexical-fallback:embed-null") → return lexical
         ├─ semantic = rankSalesBySimilarity(vec, upcomingSales)  [services/semanticSearch.ts]
         │    (findings JOIN images ON imageId
         │       WHERE images.embedding IS NOT NULL
         │         AND images.embedModel = EMBED_MODEL AND images.embedDim = EMBED_DIM
         │     → blobToFloat32 → per-row embedDim check against the query vector's length → cosine
         │     → drop sub-SEMANTIC_MIN_COSINE rows (soft floor only, see Design decisions)
         │     → best-per-sale = explicit max-by-cosine reduce: `if (!seen || score > seen.score) seen = candidate`
         │       — NOT first-encountered, since the SQL fetch order is unordered)
         ├─ fused = fuseRRF(lexical, semantic)    [lib/hybridSearch.ts]
         │    (k scales to the observed list length, default ~10, env-overridable via SEMANTIC_RRF_K —
         │     RRF's rank-based fusion, not the raw cosine value, is what actually discriminates results)
         ├─ log("hybrid", { candidateRows, cosineWallTimeMs })   — cheap per-search observability
         └─ return fused   (RankedSale[], same shape; `score` is now an RRF sum, a semantic contract change — see API contract)
```

Everything semantic is wrapped in try/catch at the orchestrator boundary so any unexpected failure (network, parse, cosine on malformed blob) degrades to the Phase 1 result rather than 500ing the request, and is logged as `lexical-fallback:<reason>` so a silently-broken Phase 2 is detectable rather than just quietly slower/worse.

## Data model
No data model changes. Phase 1 needs no new columns — it reads `finding_items.category`, which already exists (ADR 0018). Phase 2 needs no new columns either: Findings don't carry their own embedding, but each Finding has `imageId` referencing `images`, and `images.embedding`/`embedModel`/`embedDim` are already persisted at scan time (ADR 0013/0016/0017) — Phase 2 joins `findings → images` at query time and reuses `blobToFloat32` from `lib/embed.ts`. The candidate query filters `images.embedding IS NOT NULL AND images.embedModel = EMBED_MODEL AND images.embedDim = EMBED_DIM`, not just `imageId IS NOT NULL` — embeddings are a post-scan pass that can lag behind newly-added Findings, fail open to null on error, or be left over from a previously-configured model/dimension, and a plain join would silently score against stale or foreign vectors. `EMBED_DIM` is mandatory (fail-fast) for the semantic path — an unset `EMBED_DIM` currently makes `parseEmbedResponse`'s dimension check a no-op, which defeats the guard entirely; additionally, each candidate row's `images.embedDim` is checked against the query vector's actual length immediately before the cosine computation, independent of the `EMBED_DIM` filter above, as a second line of defense against a mixed-dimension corpus. Findings with a null `imageId` (pre-backfill) are simply excluded from the semantic ranking pass; they remain visible via the Phase 1 lexical path. The query-embedding cache is in-process memory, not a table.

## API / interface contract
- `/api/discover/search?q=` request/response contract is unchanged at the shape level: `{ sales: RankedSale[] }`. No new query params, no new response fields. **However, `RankedSale.score` changes meaning under Phase 2**: in Phase 1 it's a lexical hit-count, in Phase 2 (hybrid) it's an RRF-fused sum across lexical and semantic rank — same field, same type, different semantics. This is a semantic contract change, not a shape change, and any client sorting/comparing `score` across the two modes should treat it as non-comparable.
- New internal signatures:
  - `lib/thesaurus.ts`: `export function expandQuery(terms: string[]): { literalTerms: string[]; expandedTerms: string[]; categories: Category[] }` — `Category` imported from `api/src/lib/items.ts` (the existing 15-value type), not a new `FindingCategory` enum.
  - `api/src/services/discover.ts`: `export function searchSales(query: string, upcomingSales?: UpcomingSale[]): RankedSale[]` — optional second param lets `searchSalesHybrid` pass in an already-fetched sale list instead of `searchSales` re-querying.
  - `lib/embed.ts` (additive export): `export async function embedQueryText(text: string): Promise<number[] | null>` — same OpenAI `/embeddings` wire as `embedImages`, single string input, short request timeout (search is a live user request, not a batch scan job), reuses `parseEmbedResponse` for the `EMBED_DIM` guard (mandatory for this path).
  - `lib/queryEmbedCache.ts`: `export async function getCachedQueryEmbedding(query: string): Promise<number[] | null>` — never throws; enforces a 2–200 char query-length gate before calling `embedQueryText`, wraps it in try/catch, absorbs errors/timeouts, returns `null` on any failure, disabled state, or out-of-range query length.
  - `lib/hybridSearch.ts`: `export function fuseRRF(lexical: RankedSale[], semantic: RankedSale[], k?: number): RankedSale[]` — `k` defaults to the observed list length (min 10) when omitted, overridable via `SEMANTIC_RRF_K`.
  - `services/semanticSearch.ts`: `export async function searchSalesHybrid(query: string): Promise<RankedSale[]>` — becomes the route's entry point in Phase 2; fetches `upcomingSales` once and passes it to both `searchSales` and `rankSalesBySimilarity`; short-circuits to lexical-only when `SEMANTIC_SEARCH_ENABLED` is false. `searchSales` itself keeps its existing signature (plus the new optional param above) and stays the Phase 1 entry point / fallback.

## Integration points

### Phase 1
- `api/src/lib/thesaurus.ts` (NEW) — canonical→everyday-term synonym map (e.g. `sofa: ["couch", "chaise", "chaise lounge", "loveseat", "sectional", "settee", "davenport"]`) plus a `category → everyday terms` map covering all 15 `Category` values (imported from `api/src/lib/items.ts`, per ADR 0018), an alias index built the same way `ALIAS_INDEX` is built in `lexicon.ts`, and `expandQuery()`. Pure data + pure function, no DB/network access — this is what FR5/FR6/FR7 require. This file has one cross-module dependency (`items.ts` for `Category`); the "Phase 1 owns only thesaurus.ts + discover.ts" isolation claim in Approach is qualified by this, not absolute.
- `api/src/services/discover.ts` — `searchSales(query, upcomingSales?)`: call `expandQuery(terms)`; escape `%`, `_`, and `\` in each expanded term before building the LIKE pattern (an unescaped `%` currently matches every description); build the description LIKE clause over the escaped `expandedTerms` (a superset of the literal terms, so literal matching is preserved — FR4/AC10); additionally query `finding_items` for rows where `category IN (categories)` scoped to `upcomingSaleIds`, and merge those finding ids into the matched set before building `RankedSale[]`. Accepts an optional pre-fetched `upcomingSales` list so `searchSalesHybrid` doesn't force a second independent fetch. `getDiscoverData()` and the sale-text-match block are untouched.
- `api/src/db/schema.ts` — no edits; `discover.ts` imports the already-exported `findingItems` table.
- `api/src/lib/__tests__/thesaurus.test.ts` (NEW) — expansion correctness, determinism (same input → same output, no Date/Math.random), the "add settee to seating terms" acceptance case (AC3), and that `categories` values are drawn from the real `Category` type (compile-time, via `import type`).
- `api/src/services/__tests__/discover.test.ts` (NEW) — couch→sofa/chaise/loveseat/sectional (AC1, AC2), category-only match via `finding_items.category` (AC3), upcoming-scope exclusion (AC9), sale-level title/city regression (AC10), determinism (AC12), and a case asserting a literal `%` or `_` in a query does not turn into a match-everything LIKE pattern.

### Phase 2
- `api/src/lib/embed.ts` — add `embedQueryText()` alongside the existing `embedImages()`/`embedBatch()`; reuses `EMBED_API_BASE`, `EMBED_MODEL`, `parseEmbedResponse`, `embeddingEnabled()`. New short-timeout constant (e.g. `EMBED_SEARCH_TIMEOUT_MS`, default 3000ms) distinct from the 120s batch-scan timeout, since this sits on a live request path (FR12/AC5). `EMBED_DIM` must be set for the semantic path to activate at all (fail-fast, not a silent no-op guard) — see Design decisions.
- **Required smoke test** (NEW, e.g. `api/src/lib/__tests__/embedQueryText.smoke.test.ts`, run against the real configured endpoint, skipped/gated when `EMBED_API_BASE` is unset): asserts a known (matching text, image) pair scores meaningfully higher than an unrelated pair. This exists because many OpenAI-compatible SigLIP servers expose vision and text towers as separate model IDs/endpoints — sending a text string to a vision-only model can 500 (caught, fine) or silently return a right-dimension garbage vector that passes the dim guard and scores wrong. Document alongside `EMBED_MODEL` config that the serving stack must expose the SigLIP **text** tower under that model id, not just the vision tower.
- **Calibration step** (NEW, one-time, e.g. `scripts/calibrate-embed-threshold.ts`): before `SEMANTIC_MIN_COSINE` is relied on for anything beyond a soft floor, embed 20–30 known query/matching-image pairs through the real configured endpoint and inspect the resulting score distribution, then set `SEMANTIC_MIN_COSINE` from that observed distribution rather than the launch default. Re-run whenever `EMBED_MODEL` changes.
- `api/src/lib/queryEmbedCache.ts` (NEW) — `Map<string, { vec: number[]; expiresAt: number }>` keyed by normalized (`trim().toLowerCase()`) query text, TTL-based expiry, a **true LRU** eviction (touch/re-insert the entry on cache hit, not just on insert) capped at 512 entries, and a query-length gate (min 2, max 200 chars) applied before `embedQueryText` is ever called, to blunt keystroke-level embedding cost (assumes the UI debounces search input — confirm before shipping). Wraps `embedQueryText` in try/catch so callers never see a throw (FR10, AC7).
- `api/src/lib/hybridSearch.ts` (NEW) — `cosine(a: number[], b: number[]): number`, a `MIN_COSINE_SIM` threshold constant (env-overridable, `SEMANTIC_MIN_COSINE`, default `0.2` — a soft floor only, see Design decisions), an RRF `k` that scales to the observed list length (default 10, env-overridable `SEMANTIC_RRF_K`), and `fuseRRF()`.
- `api/src/services/semanticSearch.ts` (NEW) — `searchSalesHybrid()`: checks `SEMANTIC_SEARCH_ENABLED` first (kill switch); fetches `upcomingSales`/`upcomingSaleIds` once and passes them to both `searchSales()` (Phase 1 baseline/fallback) and `rankSalesBySimilarity()`, rather than each re-querying independently; short-circuits to lexical-only when disabled/timeout/dim-mismatch/error, logging `lexical-fallback:<reason>` in every short-circuit case and `hybrid` on success (required observability, so a silently-broken Phase 2 is detectable). The candidate query is `findings JOIN images` scoped to upcoming sale ids, filtered to `images.embedding IS NOT NULL AND images.embedModel = EMBED_MODEL AND images.embedDim = EMBED_DIM`; per row, `images.embedDim` is checked against the query vector length before cosine. Computes cosine per Finding via `blobToFloat32`, drops sub-threshold results, groups **best-per-sale via an explicit max-by-cosine reduce** (`if (!seen || score > seen.score) seen = candidate` — the SQL fetch is unordered, so first-encountered would be wrong) to build a semantic `RankedSale[]`, then calls `fuseRRF`. Logs candidate-row count and cosine-loop wall time per search (cheap) so corpus growth is observable rather than assumed away; optionally caps re-ranking to the Phase 1 lexical top-N sales if that count becomes large.
- `api/src/routes/discover.ts` — one-line change: the `GET /search` handler calls `searchSalesHybrid` instead of `searchSales`. Gate the embedding branch behind `SEMANTIC_SEARCH_ENABLED` (kill switch, redeploy-free disable) and flag that **no rate limiting exists anywhere in this app today** — an authenticated caller can script arbitrarily many unique queries against this endpoint to run up embedding cost; a rate limit or per-caller quota on the embedding branch specifically is a required follow-up before this ships broadly, not a nice-to-have. `GET /` (`getDiscoverData`) is untouched.
- `api/src/lib/__tests__/embed.test.ts` — extend with `embedQueryText` cases: dimension-mismatch discard (AC8), timeout → null (AC5 building block), and `EMBED_DIM` unset → semantic path refuses to activate (fail-fast, not silent).
- `api/src/lib/__tests__/queryEmbedCache.test.ts` (NEW) — call-count assertion against a mocked endpoint proving a repeated query embeds once (AC7), true-LRU behavior (a cache hit moves an entry to the back of the eviction order, unlike FIFO), the 512-entry cap, and the 2–200 char query-length gate.
- `api/src/lib/__tests__/hybridSearch.test.ts` (NEW) — RRF ordering, exact-lexical-never-suppressed (AC11), and `k` scaling with list length vs. the `SEMANTIC_RRF_K` override.
- `api/src/services/__tests__/semanticSearch.test.ts` (NEW) — `EMBED_API_BASE` unset → Phase 1 passthrough (AC4), endpoint timeout/error → Phase 1 passthrough within fallback budget (AC5), semantic-only match with no lexical overlap (AC6), candidate query excludes rows with a stale `embedModel`/`embedDim`, and a **required unit test** for the best-per-sale max-by-cosine reduce against an intentionally-unordered fixture (asserting it is not first-encountered).

## Technology choices
- **RRF (Reciprocal Rank Fusion)** for combining lexical and semantic rankings: simple (`score += 1/(k + rank)`), no tuning of relative score magnitudes between an ad-hoc lexical score and a cosine similarity (which live on incomparable scales), and it structurally guarantees anything present in the lexical list keeps a nonzero, present score — satisfying "exact matches never suppressed" (FR11/AC11) without extra bookkeeping. The web-scale-IR convention of `k=60` is nearly flat and barely discriminates on a 5–20 item result list, so `k` here scales to the observed list length (default ~10) instead, with `SEMANTIC_RRF_K` available for operators who want a fixed value.
- **JS cosine over `blobToFloat32`, no sqlite-vec / vector index**: the corpus is scoped to upcoming Sales' Findings only (small, bounded working set per ADR 0018's own precedent of in-process scoring), and adding a vector-search dependency would be a new piece of infra to operate and reason about for a query pattern that's a handful of cosine dot-products in JS. Revisit only if upcoming-sale Finding counts grow into the tens of thousands — per-search candidate-row and wall-time logging (see Integration points) makes that trigger observable instead of assumed.
- **In-process `Map`, true LRU (touch-on-hit) with a 512-entry cap, not FIFO and not a DB-backed or Redis cache**: satisfies FR10's "cache query embeddings" without a new external service (explicit operator non-negotiable); acceptable to lose the cache on process restart since the cost being avoided is per-search embedding calls within a session, not cross-deploy durability. A prior draft's "LRU-by-insertion-order" was actually FIFO (never touched on hit); corrected here, plus a size cap so a long session can't grow the map unbounded.
- **FTS5 — not adopted.** Called out and rejected: FTS5 would help ranking/relevance for large free-text corpora, but the current LIKE-scan approach is already fast enough at this corpus size, FTS5 needs a virtual-table migration + trigger-based sync with `findings`, and neither Phase's acceptance criteria requires it. Revisit if Findings volume or query latency becomes a real problem.

## Design decisions
- **`SEMANTIC_MIN_COSINE` is a soft floor, not a trusted absolute cutoff.** SigLIP is trained with a sigmoid loss (`scale·cos + bias`), so raw text-image cosine sits in a compressed, checkpoint-specific band that has no established relationship to the image-image similarity ADR 0017 validated for the frozen model. The launch default (`0.2`) is a CLIP-flavored guess, not a measured value. Discrimination between results is expected to come from RRF's rank-based fusion, not from the raw cosine number; the threshold only exists to drop obvious noise before ranking. It must be calibrated (see Integration points: calibration step) before anyone treats it as meaningful.
- **The SigLIP text tower must be verified, not assumed to exist at `EMBED_MODEL`.** Serving stacks commonly split vision and text towers across separate model ids/endpoints. A required smoke test (see Integration points) asserts a known matching pair outscores an unrelated pair; the operational doc for `EMBED_MODEL` must state that the configured id has to serve the text tower, not just vision.
- **RRF `k` scales to the observed result-list length (default ~10) instead of the web-IR convention of 60.** On a 5–20 item list, `k=60` makes `1/(k+rank)` nearly flat across all ranks and barely discriminates. `SEMANTIC_RRF_K` is available as an explicit override for operators who want a fixed value with documented rationale instead.
- **`finding_items.category` (Phase 2 match) and `discover.ts`'s regex-based `tagFinding`/`scoreFinding` (existing `tally`) are deliberately left unreconciled in Phase 1.** `finding_items` is currently write-only (nothing reads it before this feature — see `docs/CHALLENGE-2026-07-02.md` line 45), and `discover.ts` already has its own independent classifier used for the response `tally`. Phase 1 introduces a third classification (category-derived match) alongside the existing regex-derived tally. A Finding can legitimately be matched via `category="seating"` while tallied `collectible` by `tagFinding` — a visible inconsistency. **Chosen stance: keep them separate and document it** (option a) rather than attempt reconciliation now; unifying the two classifiers is out of scope for this feature and tracked as a follow-up, not silently patched over.
- **`SEMANTIC_SEARCH_ENABLED` is a required kill switch.** The hybrid path must be disableable via env var alone, with no code revert or redeploy, since it's a new live-request dependency on an external embedding endpoint.
- **Every search logs its mode.** `hybrid` on a full semantic pass, `lexical-fallback:<reason>` (disabled / embed-null / timeout / error / dim-mismatch) on every short-circuit. Without this, a permanently-broken Phase 2 degrades silently to Phase 1 forever and nobody notices.
- **Candidate rows are filtered by `embedModel`/`embedDim`, not just a bare `imageId` join.** Embeddings are a post-scan pass that can lag new Findings, fail open to null, or be left over from a previously-configured model. Filtering on `embedding IS NOT NULL AND embedModel = EMBED_MODEL AND embedDim = EMBED_DIM` prevents scoring against stale or foreign-dimension vectors; a per-row `embedDim` check against the query vector's actual length is a second, independent guard right before the cosine call.
- **`EMBED_DIM` is mandatory (fail-fast) for the semantic path**, not an optional soft check. Today, an unset `EMBED_DIM` makes `parseEmbedResponse`'s dimension guard a no-op; Phase 2 must refuse to activate rather than silently skip the guard.
- **Corpus growth is instrumented, not assumed away.** Candidate-row count and cosine-loop wall time are logged per search specifically so "revisit if the corpus grows" (Technology choices) has an observable trigger instead of being a vague future promise; re-ranking can optionally be capped to the Phase 1 lexical top-N sales if that count grows large.
- **`searchSalesHybrid` fetches `upcomingSales`/`upcomingSaleIds` once** and passes them into both `searchSales` and `rankSalesBySimilarity` as an optional parameter, instead of `semanticSearch.ts` independently re-querying. This was accepted as an easy pure refactor rather than permanent cross-phase debt.
- **`thesaurus.ts` imports `Category` from `api/src/lib/items.ts`** instead of defining a parallel `FindingCategory` type. This is a deliberate, acknowledged cross-module dependency; the "Phase 1 is isolated to thesaurus.ts + discover.ts" framing in Approach is qualified by this one import.

## Risk areas
- **`SEMANTIC_MIN_COSINE` is an unverified, checkpoint-specific constant at launch.** Until the calibration pass (Integration points) runs against the real endpoint, the default may drop good semantic matches or admit noise without erroring — RRF's rank fusion softens this but does not eliminate it, since a threshold set too high can empty the semantic list entirely.
- **Silent SigLIP text-tower corruption.** If the configured `EMBED_MODEL` only serves the vision tower, a text query embedding can come back as a right-dimension, wrong-content vector — passes the dim guard, produces plausible-looking but meaningless cosine scores, and nothing errors. The required smoke test catches this at deploy/config time but does not protect against the serving stack silently changing later; consider re-running the smoke test on a schedule or on `EMBED_MODEL`/`EMBED_API_BASE` change.
- **No rate limiting exists anywhere in this app**, and the embedding branch of `GET /api/discover/search` is a new per-request cost center. An authenticated caller can script unique queries to force fresh (uncached) embedding calls at will; this is a real cost/abuse exposure until a rate limit or quota is added on that branch specifically.
- **`finding_items.category`-derived match and regex-derived `tally` can visibly disagree** for the same Finding (see Design decisions). This is an accepted, documented inconsistency for now, not a bug to silently patch — but it is user-visible and worth a follow-up ticket to unify the two classifiers.
- **Thesaurus coverage is a manual, static list** — like `LEXICON`, it only helps queries whose everyday term was anticipated (couch, chaise, loveseat, sectional, settee for `seating`); an unanticipated synonym gets no expansion in Phase 1, and Phase 2's semantic path is the only backstop for that gap, so Phase 2 being down, disabled via kill switch, or not yet merged reintroduces the exact vocabulary-mismatch problem this feature exists to fix.
- **Timeout-based fallback (FR12/AC5) is a race against a live HTTP request inside a user-facing search call** — an `EMBED_API_BASE` that's slow-but-not-down (e.g., GPU broker queued behind other jobs) risks making every search feel slow even though it "gracefully falls back," unless the short timeout (~3s) is tuned well below user-perceived latency tolerance.
- **`finding_items.category` join for category-level expansion (FR3) adds a second query to every Phase 1 search that has a category-mapped term** — for the vast majority of everyday-object queries this is the common case, so it's not a rare-path cost; acceptable at current scale but worth a query-plan check (index on `finding_items.category` and `.saleId` already exist per schema) if search latency regresses.
- **Embedding coverage can silently shrink after an `EMBED_MODEL`/`EMBED_DIM` rotation.** Filtering candidates to the currently-configured model/dim (a deliberate fix, see Design decisions) means Findings embedded under a prior model become invisible to semantic ranking until re-embedded — correct behavior, but an operator rotating `EMBED_MODEL` should expect a temporary recall drop until the backfill catches up, and should watch the candidate-row-count log for it.
- **Corpus growth cost is now observed but not yet capped by default.** The candidate-row count and cosine wall-time log (Design decisions) makes the "revisit if the corpus grows" trigger visible, but the optional top-N cap on re-ranking is not on by default; if upcoming-sale Finding counts grow substantially, someone has to notice the log and turn the cap on.
- **Query-length gate assumes UI debounce.** The 2–200 char gate on `queryEmbedCache`/`embedQueryText` blunts keystroke-level embedding cost but is not a substitute for actual debouncing in the UI; confirm the frontend debounces search input before relying on the gate alone to control cost.
