## Changes made
- **Timeout was wrongly conflated with sidecar-unreachable — the single most significant fix.** The original design threw the same `SidecarUnreachableError` for both a genuine connect-level failure (sidecar down) and a per-call operation timeout (sidecar reachable but one page slow/hung). Since `scraper/index.ts`'s Scan loop has no try/catch around individual `fetchText` calls, a timeout-as-throw would abort the *entire remaining Scan* over one slow estatesales.net page — a real regression from today's graceful skip-and-continue behavior, and a contradiction of AC-9's "same shape of data as pre-migration." Now: only a connect-level failure (ECONNREFUSED/ECONNRESET/DNS) throws; a timeout is treated like a non-2xx status (log + null + continue). Rewrote FR-3/FR-4/FR-6/AC-3/AC-5 in requirements.md and the API contract table in plan.md.
- **Rejected-promise cache eviction**, mirroring a bug fashion-monitor's own equivalent `session.ts` hit and fixed during its own ship-it run: a failed `createContext()` must evict itself from the shared-context cache (with an identity-check guard), or a single transient sidecar 500 permanently poisons the cache for the rest of the batch with no self-heal.
- **`closeSidecarSession()` must never throw.** A `finally`-block exception replaces whatever error was already propagating in JS — if the sidecar is genuinely down (already threw `SidecarUnreachableError`), an unguarded close attempt in the cleanup `finally` would silently swap that clear error for a confusing "failed to close: ECONNREFUSED", undermining the whole point of the fail-fast design.
- **Added the missing integration point: `api/src/scan/index.ts`.** This is the *actual* production CLI entrypoint (`npm run scan` → `tsx src/scan/index.ts`) — it calls `scrapeWithinRadius()` but was absent from the original plan/steps entirely, which only referenced `scraper/index.ts` directly. Added as a new requirement, plan integration point, and steps.md Step 7, to confirm sidecar-origin errors stay attributable through its own error handling.
- **Fixed a straw-man rationale** in plan.md's session-lifecycle design: the "why not a context-per-call" argument incorrectly invoked `max_contexts` capacity (a context-per-call design would also only ever hold one slot, since closed immediately) — corrected to the real reason (amortizing context-creation cost and stealth-fingerprint consistency across a batch).
- **`client.ts` must not literally port a nonexistent helper.** Fashion-monitor's reference `client.ts` imports `fetchWithTimeout` from its own `http.ts`; estate-scraper's `http.ts` has no such export, and importing it anyway would create a circular dependency (`http.ts → session.ts → client.ts → http.ts`). Plan now states `client.ts` must implement its own inline timeout helper.
- **Steps.md restructured**: split the overloaded Step 3 into 3a/3b, folded two undersized steps (scraping.ts comment, .env.example) into their natural parent steps, fixed two falsely-serialized "Parallelizable: No" steps to Yes, un-serialized test-writing steps from implementation steps they didn't actually depend on, and added a new import-path integration test step (previously asymmetric — only the scraper path had one).

## Critiques rejected
- **"Deploy the sidecar to production" moved back in-scope** — rejected. This is an already-negotiated, cross-repo scope boundary matching fashion-monitor's identical precedent (same rollout, same accepted gap, owned by home-infra/scraper-commons). Re-litigating it here would contradict the outer rollout plan's own decision log.
- **Feature flag / kill-switch to fall back to `fetch()` during the deployment gap** — rejected as scope creep. Fashion-monitor's equivalent migration shipped without one; this is a low-stakes, weekly, home-lab batch job, not a high-availability system.
- **SIGINT/Ctrl-C cleanup handler** — rejected as scope creep beyond the stated functional requirements; documented as an accepted gap instead (the sidecar's own idle-context reaper is the eventual fallback).
- **Sidecar authentication/TLS, URL scheme/options validation duplicated client-side** — rejected; the sidecar already validates all of this server-side (confirmed directly against `app.py`: URL scheme allowlist, timeout bounds, stealth-options validation all happen before any job is submitted), and no-auth-by-design on a loopback-only service is an explicit, already-accepted constraint matching fashion-monitor's identical design.
- **Log/URL sanitization for estatesales.net query params, verbose sidecar error messages** — rejected as over-generic security boilerplate not grounded in the actual target site (no login, no session tokens involved) or the actual sidecar's error message shapes (coarse, not stack traces).
- **Cross-repo shared sidecar-client package** — rejected; requirements.md already explicitly scoped this out as a separate decision.
- **Multi-tenant sidecar capacity contention across repos** — accepted as a documented, informational risk-area note only (not a fix this feature should build for); it's a home-infra/scraper-commons deployment-time concern once/if the sidecar is shared.

## Open questions requiring human input
- None that block this feature. The one loudly-flagged, already-accepted-elsewhere fact: once merged and deployed, this migration converts the weekly Scan from "silently degrades if estatesales.net blocks plain fetch()" to "hard-fails every week until the sidecar is actually deployed reachable in production" — this is the correct, intended behavior per FR-4, and mirrors exactly what happened when fashion-monitor shipped its own equivalent migration. The production deployment gap itself remains tracked in `~/dev/home-infra/docs/plans/stealth-sidecar-rollout.md`, not this feature's to close.
